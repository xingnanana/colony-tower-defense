# 技术实现

## 技术栈

纯原生 Web 技术，单文件架构，无框架依赖。

| 层级 | 技术 |
|------|------|
| 结构 | HTML5 |
| 样式 | CSS3（Flexbox、绝对定位） |
| 逻辑 | JavaScript ES6+（无 TypeScript、无 Babel） |
| 渲染 | Canvas 2D API |
| 存储 | 浏览器 `localStorage`（设置与 10 个游戏存档槽） |
| 文件 IO | 本地开发服务器读写 `data/balance.json` |

## 架构模式

- **全局状态对象** `G`：所有运行时数据（建筑数组、居民数组、资源、相机等）
- **配置常量** `CFG`：世界尺寸、昼夜时长、速度数值等调参
- **面向对象**：`Building`、`Resident`、`Enemy`、`Particle` 四个核心类
- **游戏循环**：`requestAnimationFrame` 驱动的 `update(dt)` → `render()`
- **时间步长**：`dt` 基于 `performance.now()` 差值，上限 0.1s，乘以 `gameSpeed`
- **渲染分层**：世界空间（相机变换）→ 屏幕空间（HUD、时钟）

## 渲染系统

### 界面布局

- 页面使用三栏 CSS Grid：开发控制栏 168px、地图自适应、建筑目录 188px
- 建筑目录中当前等级已开放或等待额外条件的建筑按设施、生产、仓储、防御分区，并使用 `bld-mark` 中文单字徽标；地板、住房不显示徽标。解锁等级恰好比当前大本营高一级的预览建筑由 `buildingPanelSectionKey()` 统一移动到“X本解锁”分区，升级后再返回原功能分类。
- 顶部资源栏使用 CSS `resource-mark` 颜色和几何形状区分资源类型
- 地图右下角工具条承载林务和施工班组操作，减少对地图主体的遮挡

### 相机变换
```
ctx.save()
ctx.translate(CANVAS_W/2, CANVAS_H/2)  // 原点移到画布中心
ctx.scale(zoom, zoom)                    // 缩放
ctx.translate(-cam.x, -cam.y)            // 平移到世界坐标
// ... 绘制世界内容 ...
ctx.restore()
// ... 绘制屏幕空间UI（时钟、夜间遮罩）...
```

### 世界坐标 ↔ 屏幕坐标
```
screenToWorld(sx, sy) → { x: (sx - W/2) / zoom + cam.x, y: (sy - H/2) / zoom + cam.y }
worldToScreen(wx, wy) → { x: (wx - cam.x) * zoom + W/2, y: (wy - cam.y) * zoom + H/2 }
```

### 建筑绘制
- 占地：虚线矩形（`setLineDash([3,6])`）
- 碰撞体：实心圆（`arc`），半径 = `min(w,h)/2 - 6`
- 蓝图：由 `drawBlueprintBuilding()` 独立绘制蓝色底图、技术网格、结构线、虚线轮廓和材料进度，不复用成品建筑外观
- 建造进度：自下而上填充矩形 + 百分比文字
- 建筑主体使用 Canvas 几何细节绘制，建筑代码仅作为低对比度识别辅助
- 大本营不绘制底部容量/住户信息条；所有建筑血条仅在受伤时绘制
- 生产建筑的工人数量条仅在蓝图完成且施工倒计时结束后绘制

### 居民绘制
- 碰撞体：半透明圆（半径 7px）
- 身体：椭圆（`scale(0.5, 1.0)` + `arc(0,0,10)`）→ 5×10px 半径 = 10×20px 直径
- 方向眼、携带物指示、用餐/困倦图标
- 守卫使用盾牌轮廓，用餐/困倦使用短条和警示标记，不使用 emoji

## 碰撞系统

`resolveCollisions(r, nx, ny)` 在每次移动前调用：

1. **建筑碰撞**：农场和废墟不阻挡；生产岗位工人在 `GOING_TO_WORK`、`WORKING`、`DELIVERING_PRODUCTION_INPUT` 状态可进入自己的岗位建筑，其他单位仍被阻挡。单位起点仍在自己的工作/居住建筑碰撞圈内时允许向外移动；离开后这些建筑恢复正常障碍判定，施工目标也始终保留碰撞
   - 圆对圆：`dist < building.collisionRadius + RESIDENT_RADIUS` → 推开
2. **居民碰撞**：跳过 hidden 居民
   - 推开（50% 重叠量）+ 垂直错开（30% 重叠量）避免死锁
3. **世界边界**：`clamp(nx, RESIDENT_RADIUS, WORLD_W - RESIDENT_RADIUS)`

## 移动系统

- 直线可通行时保持连续坐标直达；受阻时使用细格网 8 方向 A*、路径压缩和局部动态避让，移动结果继续通过 `resolveCollisions` 处理建筑与居民碰撞
- 地板格提供默认 1.5 倍移动速度，由 `BLD_DEFS.floor.moveSpeedMultiplier` 配置；地板没有实体等级，等级上限固定为 1。
- 道路和旧流场系统未启用；所有可移动实体共用事件驱动的全局 A* 与停滞重算

## 昼夜系统

```
一天 = DAY_LENGTH（默认 256s）
钟面周期 = CLOCK_HOURS（12h）
夜晚区间 = NIGHT_START_HOUR → NIGHT_END_HOUR（允许跨 12:00）
白天时长 = DAY_LENGTH × 非夜晚小时数 / CLOCK_HOURS
正式夜晚时长 = DAY_LENGTH × 夜晚小时数 / CLOCK_HOURS - TRANSITION × 2
时钟时间 = (totalTime / DAY_LENGTH × CLOCK_HOURS + NIGHT_END_HOUR) % CLOCK_HOURS
```

相位转换：`day → dusk → night → dawn → day`
过渡期间 `nightAlpha` 从 0 线性过渡到 0.55

- 配置编辑器只暴露“一天时长”和双滑块“夜晚区间”。`DAY_DURATION`、`NIGHT_DURATION` 继续作为状态机的派生运行时值，不再直接保存到策划配置。
- `applyBalanceData()` 兼容旧配置中的 `DAY_DURATION` / `NIGHT_DURATION`；`BALANCE_VERSION = 2` 还会把旧版 24 小时制的夜晚和用餐时刻除以 2，保持它们在一天中的相对位置。
- `tools/dev-server.js` 的保存接口接受 `1 <= version <= 2`，避免客户端升级到版本 2 后仍被旧的 `version === 1` 校验拒绝；超过当前版本的配置继续拒绝写入。

- 调试状态 `G.debugTimeLock` 可设为 `day`、`night` 或 `null`。“永久白天”完整推进白天时钟和配置的两次用餐，到达日落后把 `totalTime` 跳过黄昏、夜晚和黎明并开始次日白天；“永久黑夜”完整推进夜晚，到达天亮后跳过黎明、白天和黄昏，清除上一夜残敌并调用 `spawnEnemyWave()` 开始下一夜。生产、移动、动画和战斗始终继续更新。
- 永久昼夜是临时开发状态，不写入游戏存档；重新开始或读取存档时统一解除并刷新按钮高亮。

## 资源系统

- 每个仓库的 `stored` 是库存事实来源，`G.resources` 是所有有效仓库的汇总缓存
- `depositToStorage` / `withdrawFromStorage` 统一更新具体库存和汇总缓存
- 专用仓库容量 50，大本营总容量 200，容量按仓库全部资源合计
- 工程师取料、居民吃饭、升级、烧制木炭和炼铁都从实际仓库扣除
- 施工班组减员会优先解除没有施工或修理目标的工程师；材料已交齐但尚未完工的建筑可被其他工程师继续认领施工。
- 工程师与所有生产/招募建筑的 `+` 共用 `getAssignableResidents()`：无岗位的 `IDLE/PATROL` 居民，以及正在执行独立伐木、狩猎、果树种植、地面物资回收或独立搬运的居民都算可分配。`residentHasFiniteIndependentTask()` 让这些居民保留当前任务；任务和携带物收尾后才正式返岗或由 `activatePendingEngineer()` 转职。产房不再使用独立的纯空闲候选池。
- 蓝图和施工点不再用 `assignedEngineer` 独占单个工程师。自动分配时先给尚无人负责的施工点和蓝图各安排至少一人；当施工任务少于工程师数量时，剩余工程师会加入已有任务。`assignedEngineer` 仅保留给当前仍采用单人占用的建筑修理任务。
- 黄昏或调试跳至夜晚时，`releaseEngineerRepairAssignment()` 同时清除工程师的修理目标和建筑的单人占用标记。修理任务只在正式白天领取，确保黎明结束后工程师能够重新发现并修复夜间受损建筑。
- `reservedConstructionMaterial()` 会把同一蓝图其他工程师已经携带、尚未交付的材料计入预留，`neededMaterials()` 只返回扣除已交付量与在途量后的真实缺口。多人可并行从仓库取料，但所有人的取料总量不会超过蓝图造价。
- 正式施工只统计已经抵达建筑工作范围、状态为 `CONSTRUCTING` 的工程师。`constructionTimer` 每帧按 `dt * 现场工程师数量` 扣减，因此两人是两倍、三人是三倍；赶路、取料和送货中的工程师不提供施工倍率。
- `buildingInteractionPoint()` 为送材料、正式施工和修理任务在目标建筑圆周外采样 24 个候选点。先选择可直达的最近点；若当前侧被相邻建筑封闭，再以 A* 路径长度选择其他可达方向。结果按目标建筑和 `navigationRevision` 缓存在工程师身上。
- `buildingInteractionRange()` 统一交付、驻场施工和修理的到达范围。仓储、吃饭、回家和守卫回营使用 `residentReachedBuilding()`：既接受进入建筑外沿容差，也接受抵达该单位缓存的 `buildingAccessTarget`。当固定圆周候选都不可用时，`buildingInteractionPoint()` 将保底点吸附到附近开放导航格，不再返回未经验证的阻挡坐标。`resolveCollisions()` 不再跳过 `buildTarget`，因此单位可在建筑外沿完成交互，但不会进入建筑圆形实体。
- `segmentHitsBuilding()` 对 `home`/`workplace` 的例外只在移动线段起点仍位于对应建筑膨胀碰撞圈内时生效，用于允许单位出生、起床或离岗后向外离开。单位已经在外部时，大本营等住所和岗位会正常成为线段障碍，避免全局路径直穿而被运行时碰撞卡住。
- `isStorage(type)` = `cat === 'storage' || type === 'town_hall'`
- `findNearestStorage(pos, resType, options)` 可按资源数量或剩余空间筛选

## 生产系统

- 生产进度在 `updateBuildings` 中每座建筑每帧结算一次
- `pendingOutput` 是生产建筑内部产物缓存，容量由 `CFG.PRODUCTION_BATCH_SIZE` 控制为 5；达到容量后由 `outputHauler` 搬运整批产物
- 批量搬运要求全部同类仓库合计拥有完整批次空间；搬运过程中可拆分存入多个仓库，且搬运居民不计入 `activeWorkersByBuilding`
- 带 `inputs` 配方的生产建筑使用 `productionInputs` 保存建筑内原料，`inputHaulers` 记录当前备料工人。每名工人通过 `productionInputReservedAmount` 最多预留并搬运 5 单位；其他工人只领取扣除已送达和在途预留后的缺口，因此整轮需求可并行搬运且不会重复领取。库存从实际取出时扣除。
- 每轮原料需求为“每份产物消耗 × 当前产物暂存上限”。整轮原料全部送达后才推进生产进度；生产每份产物时从建筑原料区扣除一份配方，产物暂存满后停止并等待整批运走。
- 护林员使用同样的 5 个批次规则连续砍树，树木不足时允许部分批次收尾
- 普通生产批次通过 `Resident.carryingFrom` 记录来源建筑；搬运途中居民被移除时，未送达部分退回建筑缓存。入库以实际存入量为准，剩余部分继续搬运。
- 黄昏不会打断携带物的搬运；黎明时仍携带物资的居民会恢复 `HAULING`，避免生产批次锁死。
- 睡眠居民在黎明阶段第一次更新时会同时切换为 `IDLE` 并设置 `hidden=false`，保证人物渲染与寻路状态在同一帧恢复，不再等到黎明过渡结束。
- 居民不再累计疲劳。黄昏和夜晚直接触发回家规则；正在搬运的居民先完成入库，随后在下一次更新中回家睡觉。
- 效率由当前处于 `WORKING` 状态的在岗工人数决定，搬运中的工人不计入
- 农场使用方形耕地美术表现和田垄细节；其可穿行规则与生产逻辑不变。
- 农场连携在建筑布局改变时缓存，避免每帧重复遍历
- 采石场和铁矿通过 `findResourceNode` 校验附近资源点，并持续生产
- 炭窑每份木炭消耗 2 木材；一级暂存上限为 5，因此工人先搬入 10 木材，再开始一轮生产，产物满 5 后运往木炭仓库。
- 炼铁机每份铁锭消耗 1 铁矿和 1 木炭；一级每轮先搬入 5 铁矿与 5 木炭。任一整轮原料未备齐时生产进度保持不变。

## 初始地图资源

- `generateResources()` 以大本营中心计算格距：树、石、铁分别要求超过 `TREE_MIN_SPAWN_RADIUS`、`STONE_MIN_SPAWN_RADIUS`、`IRON_MIN_SPAWN_RADIUS`。
- 树木先在大本营初始视野内补足 `START_VISIBLE_TREE_MIN`，之后按总数生成；石头和铁矿按簇数量生成。`sampleClusterSize()` 使用截断分布匹配树 4、石 1.5 的目标均值和对应上限；铁矿以单个为主并保留低概率小簇。
- “初始地图资源”配置分类保存全部半径与簇分布参数，重开游戏后生效；编辑器校验可见树下限不超过树木总数、树木最小半径小于大本营视野、平均簇大小不超过对应上限。

## 建筑交互点

- 生产岗位使用 `workplaceInteriorPoint()` 按同一建筑当前工人顺序生成动态内部站位。单人位于中心，多人围绕中心均匀错开，站位半径保证脚底碰撞圆保持在建筑内部。`residentCanEnterOwnWorkplace()` 只为该建筑的岗位工人在返岗、工作和送入生产原料时跳过建筑碰撞；搬运、吃饭或路过的其他单位仍被阻挡。岗位人数变化时在岗工人会以半速调整到新站位，避免多人争抢中心造成堵门。
- `Building.guardSpot()` 定义箭塔守卫的固定驻守位置。仓库取放材料、居民吃饭、回家睡眠、守卫回营，以及工程师施工与修理均通过 `setBuildingInteractionTarget()` 前往建筑圆形外沿的可达交互点；建筑中心仅用于统一的到达距离判断。单位显示和移动保持连续坐标，格网仅用于路径算法。

## 性能考虑

- Canvas 渲染为主，HTML overlay 为辅
- 地板、网格、建筑、居民、敌人、资源和粒子按相机视口裁剪
- HUD 每 150ms 更新一次，避免每帧写 DOM
- 农场连携只在放置、移动、完成或拆除时重算
- `SpatialHash` 使用 96px 空间桶；建筑与矿石索引仅在 `navigationRevision` 变化时重建，居民索引每帧重建
- 寻路线段只查询覆盖区域的障碍桶，居民碰撞只查询当前位置附近的居民桶
- HUD 在一次居民遍历中统计工程师、守卫和无家者；箭塔直接使用 `assignedGuard`，树木占用使用 `targetedTrees` 集合
- 粒子系统有生命周期自动清理

## 自动化测试

- 运行 `node --test tests/*.test.js`
- 当前覆盖批量入库、搬运工移除返还、黄昏持续搬运、多障碍绕行和路径版本失效
- 逻辑测试按浏览器顺序读取 `js/` 中的逻辑模块并在隔离上下文运行；模块加载测试再按 `game.html` 的真实 `<script src>` 顺序逐文件执行并完成游戏初始化，不需要第三方依赖

## 模块化决策

项目采用无构建步骤的经典脚本分层，既可由 3010 开发服务器加载，也保留直接打开 `game.html` 的游玩能力。脚本按依赖顺序划分为：

1. `config.js`：默认数值、建筑定义、设置和 `G` 状态。
2. `world.js`、`navigation.js`、`tasks.js`：世界查询、资源迷雾、寻路和任务公共能力。
3. `entities.js`、`save-game.js`：实体模型与持久化。
4. `simulation.js`、`combat.js`、`residents.js`、`buildings.js`、`game-controls.js`：初始化及各领域更新循环。
5. `developer-ui.js`、`render.js`、`ui.js`、`input.js`：开发工具、绘制和交互。
6. `main.js`：唯一启动入口与 `requestAnimationFrame` 主循环。

当前仍使用经典脚本共享全局词法作用域，避免一次性引入打包器和大范围导出改造；新增模块必须在 `game.html` 与 `tests/module-load.test.js` 中保持相同顺序。

## 开发者数值持久化

- `data/balance.json` 保存策划覆盖值，`js/config.js` 中的默认配置继续作为无服务器启动时的回退值。
- 大本营作为万能仓库，不绘制建筑仓储条；所有专用仓库统一使用底部容量条表达当前库存与上限，任何建筑都不再显示重复的 `BOX` 数字。
- 调试面板的“全部解锁”会同步满足建筑定义中的附加解锁条件，例如木材仓库的累计砍伐树木要求。
- 调试面板使用 `- 居民 +` 与 `- 守卫 +` 步进控件；守卫调试生成会按当前昼夜状态创建，并分配至可用的守卫住所。
- 通过 `node tools/dev-server.js` 启动本地静态服务；`GET /api/balance` 加载配置，`POST /api/balance` 校验后以临时文件替换方式写盘。
- `.vscode/tasks.json` 提供“启动游戏开发服务器”后台任务，供 VS Code 任务面板启动同一 Node 服务；内嵌预览应访问 3010 端口，不能使用 Live Preview 的静态服务器。
- `.vscode/launch.json` 使用 `editor-browser` 调试类型和上述任务作为 `preLaunchTask`，使 `F5` 自动启动服务后在 VS Code 内嵌浏览器调试游戏。
- `tools/dev-server.js` 遇到 3010 已被占用时会输出同一就绪标记并正常退出，允许 VS Code 任务复用已有服务器。
- 页面初始化游戏状态前加载覆盖值，避免初始资源、建筑定义与已保存配置不一致。
- 每次打开配置编辑器都会创建独立的 `cfgEditSession`。切换分类、项目、建筑或等级时，当前输入由 `cfgCaptureCurrentPanel()` 暂存；返回页面时由 `cfgRestoreCurrentPanel()` 恢复，关闭编辑器会丢弃尚未应用的草稿。
- “应用”会把本次会话访问并修改过的全部页面合并到完整候选配置，统一完成字段和跨字段校验后一次写入运行时；任一页面有误则整次不生效。昼夜时长和初始资源等初始化字段仍需要重开生效。
- “保存到项目”先执行上述全会话应用，再由 `collectBalanceData()` 序列化全部全局、职业、敌人、建筑和世界资源配置，而不是只保存当前页面。
- 服务仅监听 `127.0.0.1`，静态文件访问限制在项目根目录内，不提供外网写入入口。
- 非 `3010` 端口页面保存失败时，编辑器显示直达 `http://127.0.0.1:3010/game.html` 的链接，避免用户手动输入地址。

## 大本营解锁配置

- 数值编辑器的“大本营解锁”分类以建筑为行、首次可建等级为值，统一编辑 `BLD_DEFS[type].unlock`。
- 应用后调用 `updateBuildingPanel()` 立即刷新建筑栏锁定状态；保存到项目时随 `buildings` 配置写入 `data/balance.json`。
- “初始人口”分类只管理开局村民数与工程师数。大本营每级村民/守卫容量分别由 `popBonus` / `guardBonus` 管理；资源初始库存位于等级 1 的 `startResources`，各资源独立仓容由每级 `storageCaps` 管理。

## 游戏设置与快捷键

- 快捷键配置保存在浏览器 `localStorage` 的 `colony-game-settings-v1` 键中，与项目数值配置分离。
- `soundEnabled` 与 `soundVolume` 使用同一个设置记录持久化，默认分别为 `true` 和 `0.55`；设置版本升级为 4，旧设置缺少声音字段时自动使用默认值。
- `js/audio.js` 使用 Web Audio API 运行时合成短音效，不依赖外部音频文件。首次用户点击时创建或恢复 `AudioContext`，兼容浏览器的自动播放限制；不支持 Web Audio 的环境会静默跳过，不影响模拟逻辑。
- 世界音效通过 `gameSoundSpatialGain()` 按声源与相机中心距离衰减；搬运、射击、命中和受击等高频事件由 `gameAudioLastPlayed` 按类别限频，避免大量单位同时活动时叠加失真。
- `playGameSound()` 统一接收语义事件名，当前覆盖 UI 点击、蓝图放置、完工、升级、招募、砍伐、入库、箭矢、近战、建筑受击与摧毁、守卫命令和昼夜提示。模拟模块不直接操作振荡器，后续可在音频层替换真实素材。
- `CHOPPING` 状态使用既有 `chopShakeBeat` 每 `0.55` 秒触发一次 `chop`，保证树木摇晃和砍击声同步；树木生命结束时单独触发 `tree_fall`，不再把一次倒树音误作完整伐木过程。
- 每次重新绑定或恢复默认都会立即写入；页面加载时合并默认值与已保存值，兼容后续新增操作。
- 游戏设置包含右键拖动判定距离，默认 `12px`、可调范围 `4px` 至 `32px`。右键累计移动达到该值后才开始拖动，短按仍作为取消或守卫命令；鼠标中键一发生移动就立即平移镜头，松开不会触发命令。
- 快捷键录制期间会阻止游戏操作；文本输入框、下拉框和数值输入框不会触发游戏快捷键。
- 默认速度快捷键使用 `KeyboardEvent.code` 的 `Digit1`、`Digit2`、`Digit3`，对应主键盘数字键。

## 游戏存档

- `serializeGameState()` 将世界状态写入 `SAVE_VERSION=1` 的存档结构，`readSaveSlots()`/`writeSaveSlots()` 统一维护固定 10 个槽位。每槽保存日期、天数、阶段和完整状态。
- 建筑、居民、敌人、动物、资源点和地面物资分别按数组编号序列化；岗位、住房、守卫驻塔、生产搬运、工程师目标、狩猎和资源归属等对象引用保存为编号，`restoreGameState()` 创建全部实体后再统一恢复引用。
- 地板掩码、昼夜时间、相机、敌人生成队列、建筑库存、生产缓存和单位生命值会持久化。粒子、箭矢、临时命令标记、选中状态、寻路网格与空间哈希属于可重建缓存，读取时清空并重新计算。
- 读取后重新统计人口与岗位、刷新资源汇总、农场连携、迷雾和建筑目录。损坏、未知建筑或版本不兼容的存档会返回明确错误，不替换当前游戏状态。
- `restoreGameState()` 先在局部变量中构造实体、恢复引用并完成标量与地板校验，全部成功后才一次性替换 `G`；深层引用损坏不会留下半恢复世界。
- 新存档以 `floorMaskRle` 游程编码保存地板掩码，继续兼容旧存档的完整 `floorMask` 数组。
- 资源点存档省略可由 `col/row/type` 恢复的坐标以及 `alive=true`、`marked=false`、`hp=1` 等默认字段，仍可读取包含完整字段的旧存档。当前约 3000 个初始资源点的单槽快照由约 446 KB 降至约 143 KB。

## 逻辑审查与性能收口

- 黄昏和调试跳夜会调用 `releaseProductionInputTask()`，清除生产建筑的备料工人集合与预留数量。
- 住房分配排除蓝图、施工中建筑、废墟和零生命建筑。招募在对应人口达到上限时暂停且保留队列；招募建筑成为废墟时清空队列。
- 生产批次按同类仓库总空位决定是否启运，`HAULING` 每次寻找仍有空位的最近仓库，因此允许跨多个专用仓库拆分入库。
- 废墟回收先依距离写入可用仓库，剩余部分创建地面物资。敌人每帧重新校验当前建筑目标的攻击距离，建筑移动后不会继续受到远程伤害。
- 资源点按格建立惰性 `Map` 索引，生成、死亡和清理时失效；建造占地查询不再逐次扫描全部资源点。
- A* 搜索复用 `costs`、`parents`、`closed` 三个 TypedArray 缓冲区，避免每次路径请求反复分配大数组。寻路调试仅包含当前可见敌人。

## 地图镜头输入

- 右键按住并拖动可平移镜头，起点可位于地面、建筑、资源点或单位上；右键在画布外弹起时也会立即结束拖动。
- 右键按下后未移动超过设置中的拖动判定距离即松开，视为取消操作或守卫命令；没有可执行守卫命令时调用 `cancelCurrentAction()` 清除当前选择、放置或砍伐标记状态。

## 砍伐命令反馈

- 砍伐工具按钮统一使用 `.active` 表示高亮，`cancelCurrentAction()` 同时清除模式、框选数据和按钮状态。
- `drawCommandCursor()` 在画布屏幕空间跟随鼠标：标记砍伐显示斧头，取消标记显示叉号，不受镜头缩放影响。
- 建筑蓝图也是鼠标命令，但不绘制鼠标跟随图标；`clearChopCommand()` 与 `clearBlueprintCommand()` 保证同一时间最多只有一个鼠标命令处于激活状态。
- 左键只用于游戏交互，不再通过左键空地拖动镜头；右键拖动不会取消当前建筑选择或放置状态。

## 护林员任务收尾

- `findForesterTree()` 会校验护林员建筑引用；工人在砍树过程中失去工作场所时，不再访问空建筑，而是完成当前木材后进入搬运或空闲状态。
- 每次砍击节拍会写入短时 `shakeUntil` 标记驱动树木渲染；节拍间隔 0.55 秒、摆动持续 0.12 秒，树干底部为旋转轴，树影不参与摆动。
- 护林员小屋不需要工人即可自动育苗：物理空格是容量上限，`foresterGrowthSlots()` 按满员砍伐速度推导并行树苗数，树苗成长时间带轻微随机波动以错开成熟时机。
- 工人仅负责搜索所属成熟树、砍伐和批量搬运；树苗成熟后自动标记为对应小屋的采伐目标。
- 砍伐前会确认木材仓库可容纳当前批次；木材仓库已满时，护林员和无业伐木者不会继续选择树木，正在砍伐的单位会中止本次砍伐并离开砍树状态。顶栏对应资源会标红。
- 无业居民执行伐木、狩猎、果树种植、地面物资回收或独立搬运时，仍可被任意生产/招募建筑分配岗位。岗位立即占用工人数，但有限任务保持原状态；任务产物入库或落地后前往新岗位。减少岗位人数时优先撤销仍在完成外部有限任务、尚未实际到岗的居民。
- `workplaceTravelPoint()` 将内部岗位返程拆成两段：单位在建筑外时先使用 `buildingInteractionPoint()` 绕行到开放外沿，`residentReachedBuilding()` 确认抵达后再切换到 `workplaceInteriorPoint()`。这样 A* 不需要把建筑内部当作开放导航格，吃饭后返岗和生产原料送入都不会停在碰撞边界。
- 资源首次进入满仓状态时，屏幕中央显示红色“某资源已满”提示，提示向上漂浮并淡出；仓位恢复前不会重复提示。
- 专用仓库完成建造或升级时会立即刷新资源统计；资源总容量为大本营当前等级容量与所有已完成同类专用仓库容量之和。
- 顶栏资源使用“当前数量/总容量”显示，例如 `5/20`，因此新建仓库后的容量变化可直接确认。

## 居民状态图标

- `drawResidentAxeIcon()` 仅在无业居民处于 `GOING_TO_CHOP` 时绘制斧头任务图标。
- 居民处于 `GOING_TO_EAT` 或 `EATING` 时绘制食物需求图标，回家或睡眠时绘制月牙图标，替代原有黄色、蓝灰状态条；食物、睡眠、斧头按优先级互斥显示。`missedMeals > 0` 时另由 `drawResidentHungerBadge()` 绘制常驻苹果徽标并改变人物轮廓，一级为琥珀色、二级为红色，不会因居民执行其他任务而隐藏；守卫不参与该逻辑。

## 定时用餐规则

- 游戏时钟跨过 `MEAL_TIME_LUNCH` 和 `MEAL_TIME_DINNER` 时，所有非守卫居民获得一次用餐需求；12 小时制代码默认值为 6:00 和 9:00，可在全局数值配置中修改。
- 持续岗位与尚未开始的任务立即调用 `beginEating()`；`CHOPPING`、`HAULING`、`GATHERING`、`BUILDING`、`CONSTRUCTING` 会设置 `finishBeforeEating`，在任务完成节点调用 `eatAfterFinishingTask()`；例如砍完当前树后先送完手上木材，再去吃饭。
- `residentMealCost()` 返回 `1 + missedMeals`：正常餐消耗 1 食物，一级/二级饥饿最多消耗 2/3。只要总库存大于 0 就会开始进食，结算时通过 `withdrawFromAnyStorage()` 跨食物仓库扣除 `min(库存, mealCost)`。第一份食物完成当前餐，额外食物按 `foodConsumed - 1` 清除对应数量的 `missedMeals`；因此二级饥饿吃 1/2/3 食物后分别保持二级、降为一级、完全恢复。只有库存为 0 时才调用 `recordMissedMeal()` 新增一次缺餐；`mealPenaltyRecorded` 防止每帧重复累计。
- `residentHungerMultiplier()` 默认在连续缺餐 1/2 次时返回 0.75/0.5，第三次将居民标记为 `starved` 并通过统一 `removeResident()` 清理岗位、住房、携带资源和目标引用。阈值由 `HUNGER_LEVEL_ONE_MULTIPLIER`、`HUNGER_LEVEL_TWO_MULTIPLIER`、`HUNGER_DEATH_MISSED_MEALS` 配置。
- 饥饿倍率由 `moveViaFlow()` 统一应用于居民移动，并计入生产、产房招募、多人施工、工程师修理、砍树、种植和狩猎效率；守卫不吃饭且不受该倍率影响。居民字段随存档保存，旧存档由 `Resident` 默认值补齐。

## 全局寻路与连续移动

- 单位坐标和实际行走保持连续；`flowMove()` 优先检测起点到目标的直线，无遮挡时直接沿任意角度移动，不表现为逐格行走。
- 直线受阻时，`findNavigationPath()` 在 `CFG.NAV_CELL`（默认 20px）的细格网上执行 8 方向加权 A*。斜向移动禁止穿过两个正交阻挡格形成的夹角。
- 通行图使用 `Uint8Array` 缓存完整建筑、石矿和铁矿的膨胀碰撞范围；农场、废墟、树木和树苗不阻挡。废墟仍由 `isCellBlocked()` 保留原建筑占地，禁止叠放新建筑，但不会进入空间哈希、通行图、局部碰撞或敌人路线阻挡。`G.navigationRevision` 变化时整张通行图与单位旧路径同时失效。
- 搜索结果经过直线可达压缩，单位只跟随必要拐点；终点被占用或完全封闭时返回距离目标最近的可达点。单次搜索最多扩展 `CFG.NAV_EXPANSION_LIMIT`（默认 4000）个节点，避免不可达目标长时间阻塞主线程。
- 普通单位通过 `queueNavigationRequest()` 提交路径请求，`processNavigationRequests()` 每个逻辑帧最多处理 `CFG.NAV_REQUESTS_PER_TICK`（默认 2）条；等待期间使用局部绕行点继续移动。目标改变、单位状态切换或导航版本变化后，队列中的旧请求会被识别并丢弃。
- 居民、守卫和敌人共用同一套全局路径规划。`resolveCollisions()`、空间哈希和前视侧向避让仍作为局部移动层，负责单位之间的短距离避让与接触分离。
- 多选守卫移动时，`guardFormationAssignments()` 根据队伍行进方向生成方阵落点，将当前位置和目标槽位分别投影到前后轴与左右轴，再按相同顺序一一配对。前排继续前往前排、后排继续前往后排，同一纵列保持左右关系，避免就近贪心造成交叉换位。`guardMovementClusters()` 再按距离拆分最多 25 人的小队，每个小队共享从自身中心计算的主路径；无法直达主路径的个体自动回退到独立规划。
- 目标、建筑布局或资源障碍改变后清除旧路径。所有通过 `moveViaFlow()` 移动的居民、工程师、守卫、敌人和动物共用停滞检测：在 `CFG.NAV_STUCK_WINDOW`（默认 2 秒）内相对窗口起点的位移小于 `CFG.NAV_STUCK_MIN_DISTANCE`（默认 10 像素），且仍远离目标时，清除局部绕行和旧路径并强制提交一次全局寻路。抵达目标、目标变化或停在不可达目标的最近可达点时重置检测窗口。
- 守卫右键移动或攻击命令会调用 `addGuardCommandMarker()`，在实际世界落点显示绿色移动准星或红色攻击准星。标记按真实帧时间淡出，因此暂停游戏不会冻结交互反馈。
- 调试面板的寻路显示由 `G.debugShowNavigation` 控制，首次加载和重新开始时默认开启，按钮默认高亮并显示“隐藏寻路”，仍可点击关闭。白天/黎明显示居民及仍在回住所的守卫，夜晚/黄昏显示守卫和敌人。`navigationDebugPoints()` 汇总当前位置、当前局部绕行点、剩余全局路径和最终目标；等待路径队列时使用虚线，居民、守卫和敌人分别使用青、绿、红色。
- 手动守卫下达移动或攻击指令时会同步 `manualTarget`、`targetX/targetY` 与调试目标。移动完成后，三者统一收口到实际到达点；目标不可达时收口到寻路计算出的最近可达点，避免寻路线跳回旧的巡逻或工作目标。
- 手动守卫只能在当前光照范围内被点击、框选、切换操控模式和接收移动目标。迷雾点击由 `canIssueManualGuardMove()` 拒绝并显示红色命令标记；`fitGuardFormationToVisibility()` 将靠近边界的队形槽位移回可见格。执行途中若光源被摧毁，`moveManualGuardWithinVision()` 会在单位跨入迷雾前撤销命令并停在最后可见位置。
- 居民和自动守卫巡逻不再每帧按角度直接挪动。`choosePatrolTarget()` 会选择一个位于已完成建筑稳定视野内、且不落入任何建筑矩形占地及居民安全边距的短程目标；农场即使允许通行，也不会成为漫游目的地。单位通过正式寻路抵达；抵达、目标失效或巡逻计时结束后才重新选择，因此调试目标与实际移动目标保持一致。

## 两态迷雾

- `G.fogVisible` 使用与世界格子一一对应的 `Uint8Array`，每 `CFG.FOG_UPDATE_INTERVAL` 秒根据现存光源重新计算，不保存探索历史。
- 迷雾只由已完成的大本营和灯驱散，半径读取各自的 `vision` 参数。普通建筑、蓝图、地板、居民、工程师和守卫均不提供视野；单位进入或停留在某处不会改变 `G.fogVisible`。
- `drawFogOverlay()` 在世界图层的最后按视口填充不可见格，因此地形、建筑、资源、单位和效果都会被完整遮挡。
- `canPlaceBuilding()` 与地板绘制会逐格检查可见性；敌人渲染和 `updateTowers()` 的目标筛选均调用 `isWorldVisible()`。
- 空闲居民巡逻使用 `isStaticPatrolVisible()`，只认可大本营和灯提供的稳定光照范围，避免巡逻进入迷雾。
- `drawNightLights()` 先在离屏 Canvas 绘制夜幕遮罩，再用 `destination-out` 径向掩膜移除大本营和灯附近的遮罩，使底下的原始白天色彩透出。`nightLightRadius()` 返回完整对应视野半径；黄昏和黎明随夜色透明度淡入淡出，不参与迷雾计算。
- 进入夜晚时 `spawnEnemyWave()` 只执行一次 `collectFogSpawnCandidates()`：从当前视野边界外、深度在 `ENEMY_SPAWN_FOG_DEPTH` 至其后一格的迷雾带建立候选点。整夜队列固定，单位实际生成前被照亮则延迟，不会重新扫描或换点。
- 波内生成时间采用均匀分段加随机扰动，避免同一波的单位同时出现；波数、间隔、持续时间、随机度和迷雾深度均可通过配置编辑器保存。

## 定制建筑配置

- `BLD_COMMON_EDIT_FIELDS` 保存建筑共用的名称、图标、占地面积、生命、建造时间与造价；`BLD_TYPE_EDIT_FIELDS` 按建筑类型声明专属字段，配置界面仅渲染该建筑实际读取的参数。占地面积使用 `宽×高` 双输入格式，修改后需重新开始生效。大本营开局即存在，因此不显示建造时间。
- 灯的迷雾范围改由 `BLD_DEFS.lamp.vision` 管理，而不是全局配置。保存该值后会立即刷新迷雾；同一值也用于升级后的视野、蓝图覆盖圈和夜间光照半径。
- 配置编辑器的建筑下拉列表由 `sortedBuildingEntries()` 按建筑的 `unlock` 大本营等级升序生成；同等级使用定义顺序作为稳定排序。选项仍只显示建筑名称，应用解锁等级后会立即重排并保留当前建筑选中状态。
- 建筑配置顶部提供大本营解锁等级与建筑类型两组多选按钮。`cfgBuildingFilters` 保存选择，同组多个值按任一值匹配，两组之间通过 `buildingMatchesCfgFilters()` 取交集；再次点击已激活按钮取消该条件。生产类额外包含产房和练兵房，仓储类额外包含住房和兵营，原 `defense` 分类显示为塔防，其余显示为其他。
- 建筑分类隐藏通用项目下拉框，`renderCfgBuildingResults()` 将筛选结果渲染为可自动换行的建筑名称按钮。点击按钮通过 `selectCfgBuilding()` 立即切换字段并高亮当前建筑；筛选改变后保留仍匹配的当前项，否则自动选择第一项。无匹配项时按钮区与字段区显示空结果提示。
- `maxLevel` 是每种建筑的独立等级上限。`levels[level]` 保存该等级的参数覆盖值；`buildingLevelValue()` 和 `buildingRuntimeDef()` 负责合并默认升级曲线与覆盖值，升级、生产、仓储、战斗、覆盖范围和生命值都从该运行时定义读取。
- `resourceIcon()` 和 `formatResourceCost()` 统一输出资源图标和消耗 HTML；每项造价同时携带 `data-resource` 与 `data-amount`。`updateBuildingCostAffordability()` 随顶部资源栏轻量刷新建筑目录，当前库存低于某项需求时只将该资源数字标红，库存达到需求后立即恢复。建筑目录、升级、招募与运行时更新均调用统一格式，避免不同位置使用难以辨认的资源文字缩写。
- `GLOBAL_EDIT_GROUPS` 将全局数值拆为"时间与居民作息""移动与寻路""树木砍伐""果树种植与收获""猎物与狩猎"；砍伐组只管理 `TREE_CHOP_TIME` 与 `TREE_WOOD_YIELD`，果树组管理 `FRUIT_TREE_CHOP_TIME`、种植、成长和食物产量。界面只渲染和应用当前组，保存项目时仍会完整写入全部全局字段。
- 全局配置先写入候选副本并完成跨字段校验，再统一更新 `CFG`，避免同组后续字段无效时前面的值已被部分写入。
- 生产建筑的产出资源使用资源下拉框。造价、初始资源等资源表由 `parseResourceMapInput()` 统一校验合法资源键、重复项、空数量和负数；整组建筑字段全部通过后才写入目标等级。

## 箭矢投射物

- `G.projectiles` 保存箭塔发射的 `ArrowProjectile`；塔的冷却在发射时开始，伤害在箭矢抵达目标时结算。
- 箭矢每帧跟随当前可见的目标位置飞行；目标死亡、离开视野或箭矢超时后，箭矢立即销毁，不会隔着迷雾造成伤害。
- 渲染使用朝飞行方向旋转的箭杆、箭头和尾羽，投射物本身进入迷雾后同样不绘制。

## 建筑覆盖范围预览

- `buildingCoverage()` 集中定义灯/大本营视野、护林员范围、矿场资源范围和箭塔射程的圆形样式与半径；无范围的建筑返回 `null`。
- `drawBuildingCoverage()` 同时用于放置蓝图、移动建筑预览和已建建筑选中状态，避免同类范围在不同交互状态下遗漏。
- 灯的覆盖圈使用暖黄色，其他范围沿用各自系统颜色；不可放置时统一改用红色提示。

## 建筑等级效果

- 运行时属性由建筑实例的 `level` 与建筑自身升级字段共同计算，而非使用全局建筑成长值。`productionSpeedMultiplier()`、`productionBufferCapacity()`、`storageCapacity()`、`towerDamage()`、`towerRange()` 和 `buildingDamageTaken()` 是统一入口。
- 生产建筑使用 `batchSize`、`levelSpeedBonus` 和 `levelBufferBonus`；护林员建筑额外使用 `saplingGrowTime`、`growJitter`，砍伐基础时间则统一读取全局 `TREE_CHOP_TIME`。育林并行树苗数也按该全局时间和护林员岗位效率推导，避免生产与补种采用两套速度。建筑专属字段在对应生产建筑的配置页编辑。
- 仓库使用 `levelCapacityBonus`；人工箭塔使用 `levelDamageBonus`；自动箭塔使用 `levelRangeBonus`；灯使用 `levelVisionBonus`；城墙使用 `levelDamageReduction`（总上限 75%）。所有值均归属对应建筑定义并保存到 `data/balance.json.buildings`。
- 建筑菜单会显示当前等级效果与下一等级预览；已建成灯的范围预览、实际迷雾和夜间光照均使用升级后的视野半径。
- `startBuildingUpgrade()` 只创建升级任务，不立即调用 `payCost()`。升级复用 `constructCost`、`constructDelivered`、`neededMaterials()` 和工程师的 `GATHERING → BUILDING → CONSTRUCTING` 状态链；材料从仓库实际取出，多个工程师共享在途预留并行搬运。材料阶段建筑保持原等级功能，施工阶段由 `constructionTimer` 暂停建筑功能并按现场工程师效率递减，归零后才应用 `upgradeTargetLevel`。
- `constructionDuration` 统一保存蓝图建造或升级施工的初始工期，供进度表现和存档恢复使用。旧存档中已扣料的 `upgrading + upgradeProgress` 会迁移为无需再次备料的工程师施工任务。

## 敌人建筑目标

- `enemyRouteBlocker()` 将敌人到大本营的直线投影到全部存活建筑，选择路径投影最靠前、并且碰撞圆与攻击距离确实覆盖该路线的建筑。
- `enemyFacingPoint()` 根据当前目标决定渲染朝向：与守卫交战时优先面向守卫，攻击建筑时面向建筑中心，未进入攻击状态时面向行进目标。守卫触发仇恨会立即清除原建筑攻击目标，因此朝向同步切换。
- 敌人只在进入该建筑攻击范围后开始攻击；没有路线阻挡时继续向大本营移动。该规则不依赖 `G.buildings` 的数组插入顺序。
- 城墙减伤在 `buildingDamageTaken()` 中统一应用，其他建筑保持原始敌人伤害。

## 恢复塔与大本营防御

- `restoration_tower` 是 1×1 防御建筑。`findRestorationTarget()` 在 `repairRange` 内从已完成、未成为废墟的建筑和可见友军中选择生命比例最低者；恢复塔不修复自身，每次只维持一个目标，目标满血、离开范围或失效后再重新选择。
- 建筑与友军分别使用 `buildingRepairRate`、`unitRepairRate`，升级通过 `levelRepairBonus` 提升两类修复效率。当前可受伤友军是守卫，接口同时兼容未来具有 `hp/maxHp` 的其他友方单位。
- `drawRestorationBeam()` 在恢复塔与当前目标之间绘制青色脉冲维修束；蓝图、施工中或被摧毁的恢复塔不会工作。建筑放置、移动与选中时使用 `repairRange` 绘制圆形范围。
- 大本营拥有独立的 `range`、`damage`、`cooldown` 参数，复用 `updateTowers()`、实体箭矢和迷雾可见性检查自动攻击敌人，不需要守卫驻扎。蓝图、施工中和废墟建筑统一禁止开火。
- 防御建筑目录包含 `data-type="restoration_tower"` 按钮；调试“全部解锁”与正常等级筛选都通过该按钮显示恢复塔，不再出现只有配置和逻辑、没有建造入口的情况。

## 果树与狩猎

- “种植果树”是独立鼠标指令。`plantFruitTree()` 只检查当前视野、占用与边界并生成 `fruit_planting` 种植点，放置时不要求库存充足，也不扣除木材。种植点使用 `requiredWood/deliveredWood` 保存材料需求与到货数量。
- 白天无岗位、非工程师村民只领取当前库存能够完整供料的种植点。领取后依次进入 `GOING_TO_PLANT_MATERIAL`、`DELIVERING_PLANT_MATERIAL`，从实际木材仓库取料时才调用 `withdrawFromStorage()` 扣除资源，送达后再进入 `GOING_TO_PLANT`/`PLANTING`。材料不足的种植点保持等待且不占用村民。
- `reservedFruitPlantingWood()` 会扣除其他已领取任务尚未取出的木材，避免多个村民同时承诺超过实际库存的果树任务。已送达材料与 `plantProgress` 都保存在种植点上，任务中断或换人不会清零。
- 材料全部送达后，村民像建筑施工一样累计 `plantProgress`。总时间由 `FRUIT_TREE_PLANT_TIME` 控制，默认 2 秒；完成时 `completeFruitPlanting()` 才将其转换为 `fruit_sapling` 并开始独立成长计时。
- 果树种植模式使用与建筑蓝图一致的一格世界预览：当前格绘制半透明树苗及虚线占地框，`canPlantFruitTree()` 同时驱动绿色/红色反馈和点击放置。迷雾、占用或越界时不可放置；木材不足不影响放置。种植点显示材料到货数量和种植进度。
- `isResourceObstacleNode()` 将普通树木、果树种植点、树苗和成熟形态统一排除在碰撞障碍之外，小人可以穿过果树所在格。
- 普通树砍伐使用 `TREE_CHOP_TIME` 与 `TREE_WOOD_YIELD`，默认每棵耗时 3 秒并提供 5 木材；成熟果树砍伐使用 `FRUIT_TREE_CHOP_TIME` 与 `FRUIT_TREE_WOOD_YIELD`，默认耗时 3 秒、不产木材（产量 0），并随机提供 `FRUIT_TREE_FOOD_MIN..MAX` 食物。护林员小屋不再保存重复的 `chopTime`；其岗位工人的实际耗时仍会除以该建筑的等级生产效率，手工伐木使用全局基础时间。树木是不可拆分的采集单位，因此产量配置高于常规携带量时仍会整棵结算，后续由统一入库与余量落地逻辑处理。`Resident.carryQueue` 让居民先后把木材（如有）和食物运入对应专用仓库，不将不同资源混入同一携带槽。
- `G.animals` 保存野生动物。开局在大本营可见范围内生成初始猎物，之后按随机化间隔缓慢补充到上限；动物在当前清晰区域内漫游，被追近时向远离猎人的方向逃跑。
- “标记狩猎/取消狩猎”与伐木使用相同的点击或框选方式。无岗位、非工程师村民通过 `GOING_TO_HUNT` 和 `HUNTING` 领取最近的未占用猎物，击杀后把随机食物送往食物仓库；已分配生产岗位的居民不会领取狩猎任务。
- 猎物进入迷雾、死亡或被取消标记后，猎人立即释放目标并恢复空闲状态。夜晚不会领取新的狩猎任务，正在进行的有限任务仍可完成。
- 野外指令 UI 使用三行固定分组：伐木标记/取消、种植果树、狩猎标记/取消。种植果树按钮在同一行显示名称，以及由 `updateFruitPlantCommandCost()` 和 `formatResourceCost()` 生成的木材数量与木材图标；配置应用后立即刷新。工程师栏位于该面板上方，不与指令面板重叠。
- 动物造型始终保持竖直，不再参与任意角度旋转。原始造型头朝左；`Animal.facingRight` 保存左右朝向，`updateAnimalFacing()` 仅根据本帧实际水平位移切换水平镜像。主要向上或向下移动、静止时沿用上一次朝向。
- 所有资源的 `HAULING` 共用仓满卸货规则：先通过 `depositToStorage()` 尽量入库；若没有可用仓库，或部分入库后全村已无剩余空间，`dropGroundItem()` 将手中余量留在当前位置并释放生产建筑的搬运锁。附近同类、未认领的地面物资会合并，避免连续生产制造大量微小实体。生产建筑和伐木不会因仓满停工。
- 白天无携带物、非工程师村民会优先通过 `GOING_TO_PICKUP` 领取可见且已有仓储空间的地面物资；空闲村民与处于巡逻、返岗或生产状态的工人均可领取。每次只拾取 `availableGroundPickupSpace()` 允许的数量，剩余堆叠继续留在地面；所有同类在途搬运量都会作为容量预留扣除，避免多人同时拿取超出可用仓容。`groundItemInteractionPoint()` 会在掉落物位于建筑或矿物碰撞体内部时选择外侧可达交互点，居民抵达该点即可拾取，物品视觉位置不变。用餐、入夜或居民移除会释放 `claimedBy`，防止物资被失效任务永久占用。

## 敌人类型与夜袭组成

- `ENEMY_DEFS` 保存普通敌人、快速敌人与破坏者的生命、移速、伤害、体型、普通/血月出现天数和两套生成权重；`data/balance.json` 的 `enemies` 区块会覆盖默认定义。
- `pickEnemyType(day, bloodMoon)` 按当前夜晚选择对应的解锁天数与权重。血月可让快速敌人与破坏者更早、更频繁地混入夜袭。
- `isBloodMoonDay()` 由天数和 `BLOOD_MOON_INTERVAL_DAYS` 推导，不额外保存状态。生成队列在入夜快照时记录 `type` 与 `bloodMoon`；血月总数和单位生命、伤害、移速分别使用独立倍率。
- `nightHoursForDay()` 以初始夜晚区间为基准。血月当天沿用当期夜长，血月结束后的下一天开始增加 `NIGHT_GROWTH_HOURS`，并受 `NIGHT_MAX_HOURS` 限制；一天总现实时间始终保持 `DAY_LENGTH`。
- `Enemy` 实例保存自身 `damage`、`attackRange`、速度和体型；守卫受击、建筑受击和渲染均使用实例属性。破坏者通过 `enemyRouteBlocker()` 优先选取路径上的 `defense` 类建筑。

## 人口生产与驻扎

- `nursery` 和 `training_ground` 通过 `recruits` 字段共享建筑内队列逻辑：完成后从建筑中心生成村民或守卫，并自动寻找对应类型可入住的建筑。
- 产房拥有两个临时生产岗位。玩家必须从真正空闲的村民中手动安排两人；招募队列只有在两人都抵达并处于 `WORKING` 时才累计进度，缺员时保留已有进度并暂停。完成中间成员后两人继续保持 `WORKING`，仅当 `recruitQueue` 归零时调用 `releaseNurseryWorkers()` 统一释放。
- `cancelNurseryRecruit()` 每次从队尾减少一名待招募村民，并通过 `storeOrDropResources()` 全额返还该成员的招募消耗。队列仍有成员时保留当前 `recruitProgress`；取消最后一名时进度归零并立即释放工人。返还资源优先进入对应仓库，满仓部分落在产房旁成为地面物资。
- 蓝图右键菜单使用独立分支，只显示材料进度、“移动”和“删除”；`startBuildingUpgrade()` 会拒绝蓝图、废墟、施工中、已升级或达到上限的建筑，防止绕过菜单重复创建升级任务。
- `G.maxPop` 与 `G.maxGuards` 分别限制村民和守卫。`availableRecruitSlots(role)` 会连同同类型的排队数量一起计算，避免多座产房或练兵房超额排队。
- `findNearestHome(pos, isGuard)` 按身份筛选床位：村民仅能入住大本营或住房，守卫仅能入住大本营或兵营。`houseCapacity()` / `guardCapacity()` 对大本营读取当前等级的 `popBonus` / `guardBonus`，升级、调试升本、配置应用、拆除住房和读取存档后均通过容量差值或 `recalculatePopulationLimits()` 更新全局上限。旧配置的 `MAX_POP_BASE` / `MAX_GUARD_BASE` 仅用于迁移到一级大本营。
- `wakeGuardAtHome()` 在黄昏将守卫放到其大本营或兵营住所外沿，再进入驻塔或自动战斗状态。夜晚结束时 `sendGuardHomeForDay()` 令守卫退出手动控制和驻塔，进入 `GUARD_GOING_HOME` 并寻路返回个人住所；只有抵达后才切换 `GUARD_SLEEPING`、隐藏并恢复生命，不再在战场原地消失。
- 左键点击或拖拽框选可见守卫；按 `E` 会对当前选中组统一切换自动/操控模式。操控模式会解除箭塔驻守，右键短按敌人下达攻击命令、右键短按地面下达按行进方向排列且共享主路径的编队移动命令，右键拖动仍用于移动镜头。
- 每次编队移动拥有独立的 `formationCommandId`。同一指令中仍在前往槽位的守卫，会忽略已经到位队友的局部避让和单位碰撞，避免近端槽位先站满后堵住远端槽位；不同指令、居民及仍在移动的队友继续正常碰撞，全部到位后恢复常规单位碰撞。
- 守卫攻击敌人时会建立仇恨目标。敌人会追击并攻击该守卫；距离超过 `ENEMY_GUARD_LEASH` 后清除仇恨，继续按原有逻辑推进和攻击建筑。
## 调试无限资源

- 点击“无限资源”后，大本营会获得所有资源各 `9999` 的真实仓储容量与库存，包括当前等级正常容量为 `0` 的资源。
- 无限资源开启期间，逻辑帧会持续补满大本营的实体库存；工程师取料、搬运和建造均读取这份真实库存，而不是只修改顶部资源统计。
- 仓储空余量仍按容量计算，不使用无限空余量，避免生产和搬运把库存写到容量上限之外。

# 技术实现

## 技术栈

纯原生 Web 技术，单文件架构，无框架依赖。

| 层级 | 技术 |
|------|------|
| 结构 | HTML5 |
| 样式 | CSS3（Flexbox、绝对定位） |
| 逻辑 | JavaScript ES6+（无 TypeScript、无 Babel） |
| 渲染 | Canvas 2D API |
| 存储 | 运行时内存（游戏存档未实现） |
| 文件 IO | 本地开发服务器读写 `data/balance.json` |

## 架构模式

- **全局状态对象** `G`：所有运行时数据（建筑数组、居民数组、资源、相机等）
- **配置常量** `CFG`：世界尺寸、昼夜时长、速度数值等调参
- **面向对象**：`Building`、`Resident`、`Enemy`、`Particle` 四个核心类
- **游戏循环**：`requestAnimationFrame` 驱动的 `update(dt)` → `render()`
- **时间步长**：`dt` 基于 `performance.now()` 差值，上限 0.1s，乘以 `gameSpeed`
- **渲染分层**：世界空间（相机变换）→ 屏幕空间（HUD、时钟）

## 渲染系统

### 界面布局

- 页面使用三栏 CSS Grid：开发控制栏 168px、地图自适应、建筑目录 188px
- 建筑目录按设施、生产、仓储、防御分区，并使用 `bld-mark` 中文单字徽标；地板、住房不显示徽标
- 顶部资源栏使用 CSS `resource-mark` 颜色和几何形状区分资源类型
- 地图右下角工具条承载林务和施工班组操作，减少对地图主体的遮挡

### 相机变换
```
ctx.save()
ctx.translate(CANVAS_W/2, CANVAS_H/2)  // 原点移到画布中心
ctx.scale(zoom, zoom)                    // 缩放
ctx.translate(-cam.x, -cam.y)            // 平移到世界坐标
// ... 绘制世界内容 ...
ctx.restore()
// ... 绘制屏幕空间UI（时钟、夜间遮罩）...
```

### 世界坐标 ↔ 屏幕坐标
```
screenToWorld(sx, sy) → { x: (sx - W/2) / zoom + cam.x, y: (sy - H/2) / zoom + cam.y }
worldToScreen(wx, wy) → { x: (wx - cam.x) * zoom + W/2, y: (wy - cam.y) * zoom + H/2 }
```

### 建筑绘制
- 占地：虚线矩形（`setLineDash([3,6])`）
- 碰撞体：实心圆（`arc`），半径 = `min(w,h)/2 - 6`
- 蓝图：`globalAlpha = 0.5`
- 建造进度：自下而上填充矩形 + 百分比文字
- 建筑主体使用 Canvas 几何细节绘制，建筑代码仅作为低对比度识别辅助
- 大本营不绘制底部容量/住户信息条；所有建筑血条仅在受伤时绘制
- 生产建筑的工人数量条仅在蓝图完成且施工倒计时结束后绘制

### 居民绘制
- 碰撞体：半透明圆（半径 7px）
- 身体：椭圆（`scale(0.5, 1.0)` + `arc(0,0,10)`）→ 5×10px 半径 = 10×20px 直径
- 方向眼、携带物指示、用餐/困倦图标
- 守卫使用盾牌轮廓，用餐/困倦使用短条和警示标记，不使用 emoji

## 碰撞系统

`resolveCollisions(r, nx, ny)` 在每次移动前调用：

1. **建筑碰撞**：跳过道路、仓库/大本营（`isStorage`）、自己工作/居住建筑
   - 圆对圆：`dist < building.collisionRadius + RESIDENT_RADIUS` → 推开
2. **居民碰撞**：跳过 hidden 居民
   - 推开（50% 重叠量）+ 垂直错开（30% 重叠量）避免死锁
3. **世界边界**：`clamp(nx, RESIDENT_RADIUS, WORLD_W - RESIDENT_RADIUS)`

## 移动系统

- 当前采用目标点直线移动，通过 `resolveCollisions` 处理建筑和居民碰撞
- 地板格提供默认 1.5 倍移动速度，由 `BLD_DEFS.floor.moveSpeedMultiplier` 配置；地板没有实体等级，等级上限固定为 1。
- A*/流场和道路系统当前均未启用
- 复杂建筑布局可能出现绕障失败，是后续需要单独设计的系统

## 昼夜系统

```
一天 = DAY_DURATION(162s) + TRANSITION(8s) + NIGHT_DURATION(78s) + TRANSITION(8s) = 256s
时钟时间 = (totalTime / 256 * 24 + 6) % 24  // 6:00 AM 起始
```

相位转换：`day → dusk → night → dawn → day`
过渡期间 `nightAlpha` 从 0 线性过渡到 0.55

## 资源系统

- 每个仓库的 `stored` 是库存事实来源，`G.resources` 是所有有效仓库的汇总缓存
- `depositToStorage` / `withdrawFromStorage` 统一更新具体库存和汇总缓存
- 专用仓库容量 50，大本营总容量 200，容量按仓库全部资源合计
- 工程师取料、居民吃饭、升级和炼铁都从实际仓库扣除
- 施工班组减员会优先解除没有施工或修理目标的工程师；材料已交齐但尚未完工的建筑可被其他工程师继续认领施工。
- 蓝图和施工点不再用 `assignedEngineer` 独占单个工程师。自动分配时先给尚无人负责的施工点和蓝图各安排至少一人；当施工任务少于工程师数量时，剩余工程师会加入已有任务。`assignedEngineer` 仅保留给当前仍采用单人占用的建筑修理任务。
- `reservedConstructionMaterial()` 会把同一蓝图其他工程师已经携带、尚未交付的材料计入预留，`neededMaterials()` 只返回扣除已交付量与在途量后的真实缺口。多人可并行从仓库取料，但所有人的取料总量不会超过蓝图造价。
- 正式施工只统计已经抵达建筑工作范围、状态为 `CONSTRUCTING` 的工程师。`constructionTimer` 每帧按 `dt * 现场工程师数量` 扣减，因此两人是两倍、三人是三倍；赶路、取料和送货中的工程师不提供施工倍率。
- `isStorage(type)` = `cat === 'storage' || type === 'town_hall'`
- `findNearestStorage(pos, resType, options)` 可按资源数量或剩余空间筛选

## 生产系统

- 生产进度在 `updateBuildings` 中每座建筑每帧结算一次
- `pendingOutput` 是生产建筑内部产物缓存，容量由 `CFG.PRODUCTION_BATCH_SIZE` 控制为 5；达到容量后由 `outputHauler` 搬运整批产物
- 批量搬运要求目标仓库一次性拥有完整批次空间；搬运中的居民不计入 `activeWorkersByBuilding`
- 护林员使用同样的 5 个批次规则连续砍树，树木不足时允许部分批次收尾
- 普通生产批次通过 `Resident.carryingFrom` 记录来源建筑；搬运途中居民被移除时，未送达部分退回建筑缓存。入库以实际存入量为准，剩余部分继续搬运。
- 黄昏不会打断携带物的搬运；黎明时仍携带物资的居民会恢复 `HAULING`，避免生产批次锁死。
- 睡眠居民在黎明阶段第一次更新时会同时切换为 `IDLE`、清零疲劳并设置 `hidden=false`，保证人物渲染与寻路状态在同一帧恢复，不再等到黎明过渡结束。
- 效率由当前处于 `WORKING` 状态的在岗工人数决定，搬运中的工人不计入
- 农场使用方形耕地美术表现和田垄细节；其可穿行规则与生产逻辑不变。
- 农场连携在建筑布局改变时缓存，避免每帧重复遍历
- 采石场和铁矿通过 `findResourceNode` 校验附近资源点，并持续生产
- 炼铁机配方为 1 铁矿生产 1 铁锭

## 初始地图资源

- `generateResources()` 以大本营中心计算格距：树、石、铁分别要求超过 `TREE_MIN_SPAWN_RADIUS`、`STONE_MIN_SPAWN_RADIUS`、`IRON_MIN_SPAWN_RADIUS`。
- 树木先在大本营初始视野内补足 `START_VISIBLE_TREE_MIN`，之后按总数生成；石头和铁矿按簇数量生成。`sampleClusterSize()` 使用截断分布匹配树 4、石 1.5 的目标均值和对应上限；铁矿以单个为主并保留低概率小簇。
- “初始地图资源”配置分类保存全部半径与簇分布参数，重开游戏后生效；编辑器校验可见树下限不超过树木总数、树木最小半径小于大本营视野、平均簇大小不超过对应上限。

## 建筑交互点

- 生产工人前往生产建筑中心附近并通过建筑碰撞半径判定到岗；农场可被单位穿过，农场工人的到岗距离为普通生产建筑的一半。
- `Building.guardSpot()` 定义箭塔守卫的固定驻守位置；生产、仓储和施工目标使用建筑中心。以上目标点仍由现有连续移动和碰撞系统处理，不引入网格化行走。

## 性能考虑

- Canvas 渲染为主，HTML overlay 为辅
- 地板、网格、建筑、居民、敌人、资源和粒子按相机视口裁剪
- HUD 每 150ms 更新一次，避免每帧写 DOM
- 农场连携只在放置、移动、完成或拆除时重算
- `SpatialHash` 使用 96px 空间桶；建筑与矿石索引仅在 `navigationRevision` 变化时重建，居民索引每帧重建
- 寻路线段只查询覆盖区域的障碍桶，居民碰撞只查询当前位置附近的居民桶
- HUD 在一次居民遍历中统计工程师、守卫和无家者；箭塔直接使用 `assignedGuard`，树木占用使用 `targetedTrees` 集合
- 粒子系统有生命周期自动清理

## 模块化决策

当前保留单文件架构，以维持直接打开 `game.html` 的运行方式。ES Module 拆分需要本地 HTTP 服务或构建流程，待项目正式引入启动脚本后再执行，避免本轮性能优化改变交付方式。

## 开发者数值持久化

- `data/balance.json` 保存策划覆盖值，`game.html` 内联配置继续作为无服务器启动时的默认值。
- 大本营作为万能仓库，但不在建筑头顶显示 `BOX` 库存提示；木材仓库改用底部进度条表达库存，不显示 `BOX` 数字。
- 调试面板的“全部解锁”会同步满足建筑定义中的附加解锁条件，例如木材仓库的累计砍伐树木要求。
- 调试面板使用 `- 居民 +` 与 `- 守卫 +` 步进控件；守卫调试生成会按当前昼夜状态创建，并分配至可用的守卫住所。
- 通过 `node tools/dev-server.js` 启动本地静态服务；`GET /api/balance` 加载配置，`POST /api/balance` 校验后以临时文件替换方式写盘。
- `.vscode/tasks.json` 提供“启动游戏开发服务器”后台任务，供 VS Code 任务面板启动同一 Node 服务；内嵌预览应访问 3010 端口，不能使用 Live Preview 的静态服务器。
- `.vscode/launch.json` 使用 `editor-browser` 调试类型和上述任务作为 `preLaunchTask`，使 `F5` 自动启动服务后在 VS Code 内嵌浏览器调试游戏。
- `tools/dev-server.js` 遇到 3010 已被占用时会输出同一就绪标记并正常退出，允许 VS Code 任务复用已有服务器。
- 页面初始化游戏状态前加载覆盖值，避免初始资源、建筑定义与已保存配置不一致。
- 编辑器“应用”负责运行时预览，“保存到项目”负责持久化；昼夜时长和初始资源等初始化字段需要重开生效。
- 服务仅监听 `127.0.0.1`，静态文件访问限制在项目根目录内，不提供外网写入入口。
- 非 `3010` 端口页面保存失败时，编辑器显示直达 `http://127.0.0.1:3010/game.html` 的链接，避免用户手动输入地址。

## 大本营解锁配置

- 数值编辑器的“大本营解锁”分类以建筑为行、首次可建等级为值，统一编辑 `BLD_DEFS[type].unlock`。
- 应用后调用 `updateBuildingPanel()` 立即刷新建筑栏锁定状态；保存到项目时随 `buildings` 配置写入 `data/balance.json`。
- “初始人口”分类只管理开局人口与人口上限。资源初始库存迁移到 `BLD_DEFS.town_hall.startResources`，各资源独立仓容由 `storageCaps` 管理，均在大本营等级 1 的建筑配置页编辑。

## 游戏设置与快捷键

- 快捷键配置保存在浏览器 `localStorage` 的 `colony-game-settings-v1` 键中，与项目数值配置分离。
- 每次重新绑定或恢复默认都会立即写入；页面加载时合并默认值与已保存值，兼容后续新增操作。
- 游戏设置包含右键拖动判定距离，默认 `12px`、可调范围 `4px` 至 `32px`。右键累计移动达到该值后才开始拖动，短按仍作为取消或守卫命令；鼠标中键一发生移动就立即平移镜头，松开不会触发命令。
- 快捷键录制期间会阻止游戏操作；文本输入框、下拉框和数值输入框不会触发游戏快捷键。
- 默认速度快捷键使用 `KeyboardEvent.code` 的 `Digit1`、`Digit2`、`Digit3`，对应主键盘数字键。

## 地图镜头输入

- 右键按住并拖动可平移镜头，起点可位于地面、建筑、资源点或单位上；右键在画布外弹起时也会立即结束拖动。
- 右键按下后未移动超过设置中的拖动判定距离即松开，视为取消操作或守卫命令；没有可执行守卫命令时调用 `cancelCurrentAction()` 清除当前选择、放置或砍伐标记状态。

## 砍伐命令反馈

- 砍伐工具按钮统一使用 `.active` 表示高亮，`cancelCurrentAction()` 同时清除模式、框选数据和按钮状态。
- `drawCommandCursor()` 在画布屏幕空间跟随鼠标：标记砍伐显示斧头，取消标记显示叉号，不受镜头缩放影响。
- 建筑蓝图也是鼠标命令，但不绘制鼠标跟随图标；`clearChopCommand()` 与 `clearBlueprintCommand()` 保证同一时间最多只有一个鼠标命令处于激活状态。
- 左键只用于游戏交互，不再通过左键空地拖动镜头；右键拖动不会取消当前建筑选择或放置状态。

## 护林员任务收尾

- `findForesterTree()` 会校验护林员建筑引用；工人在砍树过程中失去工作场所时，不再访问空建筑，而是完成当前木材后进入搬运或空闲状态。
- 每次砍击节拍会写入短时 `shakeUntil` 标记驱动树木渲染；节拍间隔 0.55 秒、摆动持续 0.12 秒，树干底部为旋转轴，树影不参与摆动。
- 护林员小屋不需要工人即可自动育苗：物理空格是容量上限，`foresterGrowthSlots()` 按满员砍伐速度推导并行树苗数，树苗成长时间带轻微随机波动以错开成熟时机。
- 工人仅负责搜索所属成熟树、砍伐和批量搬运；树苗成熟后自动标记为对应小屋的采伐目标。
- 砍伐前会确认木材仓库可容纳当前批次；木材仓库已满时，护林员和无业伐木者不会继续选择树木，正在砍伐的单位会中止本次砍伐并离开砍树状态。顶栏对应资源会标红。
- 无业居民正在前往手工砍树、砍伐或运送该木材时，仍可被生产建筑分配岗位。岗位立即占用工人数，但 `finishCurrentChopForWork` 会保留当前任务；居民只完成手头这一棵树并将木材送入仓库，随后前往新岗位，不会因被分配至护林员小屋而继续砍下一棵。
- 资源首次进入满仓状态时，屏幕中央显示红色“某资源已满”提示，提示向上漂浮并淡出；仓位恢复前不会重复提示。
- 专用仓库完成建造或升级时会立即刷新资源统计；资源总容量为大本营当前等级容量与所有已完成同类专用仓库容量之和。
- 顶栏资源使用“当前数量/总容量”显示，例如 `5/20`，因此新建仓库后的容量变化可直接确认。

## 居民状态图标

- `drawResidentAxeIcon()` 仅在无业居民处于 `GOING_TO_CHOP` 时绘制斧头任务图标。
- 居民处于 `GOING_TO_EAT` 或 `EATING` 时绘制食物需求图标，回家或睡眠时绘制月牙图标，替代原有黄色、蓝灰状态条；食物、睡眠、斧头按优先级互斥显示。

## 定时用餐规则

- 游戏时钟跨过 `MEAL_TIME_LUNCH` 和 `MEAL_TIME_DINNER` 时，所有非守卫居民获得一次用餐需求；默认时刻为 12:00 和 18:00，可在全局数值配置中修改。
- 持续岗位与尚未开始的任务立即调用 `beginEating()`；`CHOPPING`、`HAULING`、`GATHERING`、`BUILDING`、`CONSTRUCTING` 会设置 `finishBeforeEating`，在任务完成节点调用 `eatAfterFinishingTask()`；例如砍完当前树后先送完手上木材，再去吃饭。
- 用餐成功时消耗 1 食物。系统不再维护饥饿值、饥饿减速或饿死惩罚。

## 全局寻路与连续移动

- 单位坐标和实际行走保持连续；`flowMove()` 优先检测起点到目标的直线，无遮挡时直接沿任意角度移动，不表现为逐格行走。
- 直线受阻时，`findNavigationPath()` 在 `CFG.NAV_CELL`（默认 20px）的细格网上执行 8 方向加权 A*。斜向移动禁止穿过两个正交阻挡格形成的夹角。
- 通行图使用 `Uint8Array` 缓存建筑、废墟、石矿和铁矿的膨胀碰撞范围；农场、树木和树苗不阻挡。`G.navigationRevision` 变化时整张通行图与单位旧路径同时失效。
- 搜索结果经过直线可达压缩，单位只跟随必要拐点；终点被占用或完全封闭时返回距离目标最近的可达点。单次搜索最多扩展 `CFG.NAV_EXPANSION_LIMIT`（默认 4000）个节点，避免不可达目标长时间阻塞主线程。
- 普通单位通过 `queueNavigationRequest()` 提交路径请求，`processNavigationRequests()` 每个逻辑帧最多处理 `CFG.NAV_REQUESTS_PER_TICK`（默认 2）条；等待期间使用局部绕行点继续移动。目标改变、单位状态切换或导航版本变化后，队列中的旧请求会被识别并丢弃。
- 居民、守卫和敌人共用同一套全局路径规划。`resolveCollisions()`、空间哈希和前视侧向避让仍作为局部移动层，负责单位之间的短距离避让与接触分离。
- 多选守卫移动时，`guardFormationAssignments()` 根据队伍行进方向生成方阵落点，将当前位置和目标槽位分别投影到前后轴与左右轴，再按相同顺序一一配对。前排继续前往前排、后排继续前往后排，同一纵列保持左右关系，避免就近贪心造成交叉换位。`guardMovementClusters()` 再按距离拆分最多 25 人的小队，每个小队共享从自身中心计算的主路径；无法直达主路径的个体自动回退到独立规划。
- 目标、建筑布局或资源障碍改变后清除旧路径；连续卡住超过 `CFG.NAV_STUCK_TIMEOUT`（默认 0.5 秒）时重新规划。
- 守卫右键移动或攻击命令会调用 `addGuardCommandMarker()`，在实际世界落点显示绿色移动准星或红色攻击准星。标记按真实帧时间淡出，因此暂停游戏不会冻结交互反馈。
- 调试面板的寻路显示由 `G.debugShowNavigation` 控制，首次加载和重新开始时默认开启，按钮默认高亮并显示“隐藏寻路”，仍可点击关闭。白天/黎明只显示居民，夜晚/黄昏只显示守卫和敌人。`navigationDebugPoints()` 汇总当前位置、当前局部绕行点、剩余全局路径和最终目标；等待路径队列时使用虚线，居民、守卫和敌人分别使用青、绿、红色。
- 手动守卫下达移动或攻击指令时会同步 `manualTarget`、`targetX/targetY` 与调试目标。移动完成后，三者统一收口到实际到达点；目标不可达时收口到寻路计算出的最近可达点，避免寻路线跳回旧的巡逻或工作目标。
- 居民和自动守卫巡逻不再每帧按角度直接挪动。`choosePatrolTarget()` 会选择一个位于已完成建筑稳定视野内、且不落入任何建筑矩形占地及居民安全边距的短程目标；农场即使允许通行，也不会成为漫游目的地。单位通过正式寻路抵达；抵达、目标失效或巡逻计时结束后才重新选择，因此调试目标与实际移动目标保持一致。

## 两态迷雾

- `G.fogVisible` 使用与世界格子一一对应的 `Uint8Array`，每 `CFG.FOG_UPDATE_INTERVAL` 秒重新计算，不保存探索历史。
- 已完成的大本营、灯和普通建筑分别提供 8 格、5 格和自身外接圆的圆形视野；蓝图和地板不提供视野。居民和守卫每帧位置参与计算，半径分别为 2 格、3 格。
- `drawFogOverlay()` 在世界图层的最后按视口填充不可见格，因此地形、建筑、资源、单位和效果都会被完整遮挡。
- `canPlaceBuilding()` 与地板绘制会逐格检查可见性；敌人渲染和 `updateTowers()` 的目标筛选均调用 `isWorldVisible()`。
- 空闲居民巡逻使用 `isStaticPatrolVisible()`，只认可已完成建筑提供的稳定视野，不认可居民、守卫等移动单位的动态视野，避免巡逻行为持续开拓迷雾。
- `drawNightLights()` 先在离屏 Canvas 绘制夜幕遮罩，再用 `destination-out` 径向掩膜移除大本营和灯附近的遮罩，使底下的原始白天色彩透出。`nightLightRadius()` 返回完整对应视野半径；黄昏和黎明随夜色透明度淡入淡出，不参与迷雾计算。
- 进入夜晚时 `spawnEnemyWave()` 只执行一次 `collectFogSpawnCandidates()`：从当前视野边界外、深度在 `ENEMY_SPAWN_FOG_DEPTH` 至其后一格的迷雾带建立候选点。整夜队列固定，单位实际生成前被照亮则延迟，不会重新扫描或换点。
- 波内生成时间采用均匀分段加随机扰动，避免同一波的单位同时出现；波数、间隔、持续时间、随机度和迷雾深度均可通过配置编辑器保存。

## 定制建筑配置

- `BLD_COMMON_EDIT_FIELDS` 保存建筑共用的名称、图标、生命、建造时间与造价；`BLD_TYPE_EDIT_FIELDS` 按建筑类型声明专属字段，配置界面仅渲染该建筑实际读取的参数。大本营开局即存在，因此不显示建造时间。
- 灯的迷雾范围改由 `BLD_DEFS.lamp.vision` 管理，而不是全局配置。保存该值后会立即刷新迷雾；同一值也用于升级后的视野、蓝图覆盖圈和夜间光照半径。
- `maxLevel` 是每种建筑的独立等级上限。`levels[level]` 保存该等级的参数覆盖值；`buildingLevelValue()` 和 `buildingRuntimeDef()` 负责合并默认升级曲线与覆盖值，升级、生产、仓储、战斗、覆盖范围和生命值都从该运行时定义读取。
- `resourceIcon()` 和 `formatResourceCost()` 统一输出资源图标和消耗 HTML；建筑目录、升级、招募与运行时更新均调用该格式化函数，避免不同位置使用难以辨认的资源文字缩写。

## 箭矢投射物

- `G.projectiles` 保存箭塔发射的 `ArrowProjectile`；塔的冷却在发射时开始，伤害在箭矢抵达目标时结算。
- 箭矢每帧跟随当前可见的目标位置飞行；目标死亡、离开视野或箭矢超时后，箭矢立即销毁，不会隔着迷雾造成伤害。
- 渲染使用朝飞行方向旋转的箭杆、箭头和尾羽，投射物本身进入迷雾后同样不绘制。

## 建筑覆盖范围预览

- `buildingCoverage()` 集中定义灯/大本营视野、护林员范围、矿场资源范围和箭塔射程的圆形样式与半径；无范围的建筑返回 `null`。
- `drawBuildingCoverage()` 同时用于放置蓝图、移动建筑预览和已建建筑选中状态，避免同类范围在不同交互状态下遗漏。
- 灯的覆盖圈使用暖黄色，其他范围沿用各自系统颜色；不可放置时统一改用红色提示。

## 建筑等级效果

- 运行时属性由建筑实例的 `level` 与建筑自身升级字段共同计算，而非使用全局建筑成长值。`productionSpeedMultiplier()`、`productionBufferCapacity()`、`storageCapacity()`、`towerDamage()`、`towerRange()` 和 `buildingDamageTaken()` 是统一入口。
- 生产建筑使用 `batchSize`、`levelSpeedBonus` 和 `levelBufferBonus`；护林员额外使用 `chopTime`、`saplingGrowTime`、`growJitter`。这些字段在对应生产建筑的配置页编辑。
- 仓库使用 `levelCapacityBonus`；人工箭塔使用 `levelDamageBonus`；自动箭塔使用 `levelRangeBonus`；灯使用 `levelVisionBonus`；城墙使用 `levelDamageReduction`（总上限 75%）。所有值均归属对应建筑定义并保存到 `data/balance.json.buildings`。
- 建筑菜单会显示当前等级效果与下一等级预览；已建成灯的范围预览、实际迷雾和夜间光照均使用升级后的视野半径。

## 敌人建筑目标

- `enemyRouteBlocker()` 将敌人到大本营的直线投影到全部存活建筑，选择路径投影最靠前、并且碰撞圆与攻击距离确实覆盖该路线的建筑。
- `enemyFacingPoint()` 根据当前目标决定渲染朝向：与守卫交战时优先面向守卫，攻击建筑时面向建筑中心，未进入攻击状态时面向行进目标。守卫触发仇恨会立即清除原建筑攻击目标，因此朝向同步切换。
- 敌人只在进入该建筑攻击范围后开始攻击；没有路线阻挡时继续向大本营移动。该规则不依赖 `G.buildings` 的数组插入顺序。
- 城墙减伤在 `buildingDamageTaken()` 中统一应用，其他建筑保持原始敌人伤害。

## 恢复塔与大本营防御

- `restoration_tower` 是 1×1 防御建筑。`findRestorationTarget()` 在 `repairRange` 内从已完成、未成为废墟的建筑和可见友军中选择生命比例最低者；恢复塔不修复自身，每次只维持一个目标，目标满血、离开范围或失效后再重新选择。
- 建筑与友军分别使用 `buildingRepairRate`、`unitRepairRate`，升级通过 `levelRepairBonus` 提升两类修复效率。当前可受伤友军是守卫，接口同时兼容未来具有 `hp/maxHp` 的其他友方单位。
- `drawRestorationBeam()` 在恢复塔与当前目标之间绘制青色脉冲维修束；蓝图、施工中或被摧毁的恢复塔不会工作。建筑放置、移动与选中时使用 `repairRange` 绘制圆形范围。
- 大本营拥有独立的 `range`、`damage`、`cooldown` 参数，复用 `updateTowers()`、实体箭矢和迷雾可见性检查自动攻击敌人，不需要守卫驻扎。蓝图、施工中和废墟建筑统一禁止开火。
- 防御建筑目录包含 `data-type="restoration_tower"` 按钮；调试“全部解锁”与正常等级筛选都通过该按钮显示恢复塔，不再出现只有配置和逻辑、没有建造入口的情况。

## 果树与狩猎

- “种植果树”是独立鼠标指令。`plantFruitTree()` 逐格检查当前视野、占用和木材库存，扣除 `FRUIT_TREE_WOOD_COST` 后只生成 `fruit_planting` 种植点，不会立即生成树苗。
- 白天无岗位、非工程师村民通过 `GOING_TO_PLANT`/`PLANTING` 自动领取最近的未占用种植点，抵达后像建筑施工一样累计种植点自身的 `plantProgress`，进度条显示完成比例。总时间由 `FRUIT_TREE_PLANT_TIME` 控制，默认 2 秒；任务中断或换人不会清零。完成时 `completeFruitPlanting()` 才将其转换为 `fruit_sapling` 并开始独立的随机成长计时；没有空闲村民时种植进度暂停。
- 果树种植模式使用与建筑蓝图一致的一格世界预览：当前格绘制半透明树苗及虚线占地框，`canPlantFruitTree()` 同时驱动绿色/红色反馈和点击放置。迷雾、占用、越界或木材不足都会显示不可放置；预览本身不扣资源。
- `isResourceObstacleNode()` 将普通树木、果树种植点、树苗和成熟形态统一排除在碰撞障碍之外，小人可以穿过果树所在格。
- 果树砍伐复用居民伐木状态机。成熟果树固定提供 1 木材，并随机提供 `FRUIT_TREE_FOOD_MIN..MAX` 食物；`Resident.carryQueue` 让居民先后把两类资源送入对应专用仓库，不将不同资源混入同一携带槽。
- `G.animals` 保存野生动物。开局在大本营可见范围内生成初始猎物，之后按随机化间隔缓慢补充到上限；动物在当前清晰区域内漫游，被追近时向远离猎人的方向逃跑。
- “标记狩猎/取消狩猎”与伐木使用相同的点击或框选方式。无岗位、非工程师村民通过 `GOING_TO_HUNT` 和 `HUNTING` 领取最近的未占用猎物，击杀后把随机食物送往食物仓库；已分配生产岗位的居民不会领取狩猎任务。
- 猎物进入迷雾、死亡或被取消标记后，猎人立即释放目标并恢复空闲状态。夜晚不会领取新的狩猎任务，正在进行的有限任务仍可完成。
- 野外指令 UI 使用三行固定分组：伐木标记/取消、种植果树、狩猎标记/取消。工程师栏位于该面板上方，不与增高后的指令面板重叠。
- 动物造型始终保持竖直，不再参与任意角度旋转。原始造型头朝左；`Animal.facingRight` 保存左右朝向，`updateAnimalFacing()` 仅根据本帧实际水平位移切换水平镜像。主要向上或向下移动、静止时沿用上一次朝向。
- 狩猎产出的食物设置 `dropCarryingWhenBlocked`。猎人进入 `HAULING` 后只要求目标仓库至少有 1 点空间；到达后通过 `depositToStorage()` 尽量入库，若全村已无剩余空间，`dropGroundItem()` 只把手中剩余数量留在当前位置并让猎人恢复空闲。搬运途中容量被其他任务填满时同样立即落地剩余食物。
- 白天无岗位、非工程师村民会优先通过 `GOING_TO_PICKUP` 领取可见且已有仓储空间的地面物资。每次只拾取 `availableGroundPickupSpace()` 允许的数量，剩余堆叠继续留在地面；在途食物会作为容量预留扣除，避免多人同时拿取超出可用仓容。用餐、疲劳、入夜或居民移除会释放 `claimedBy`，防止物资被失效任务永久占用。

## 敌人类型与夜袭组成

- `ENEMY_DEFS` 保存普通敌人、快速敌人与破坏者的生命、移速、伤害、体型、出现天数和生成权重；`data/balance.json` 的 `enemies` 区块会覆盖默认定义。
- `pickEnemyType(day)` 仅从已到达 `unlockDay` 的单位中按 `spawnWeight` 抽取。普通敌人始终可用，快速敌人默认第 3 天加入，破坏者默认第 6 天加入。
- 敌人生成队列在入夜快照时记录 `type`，因此同一夜的单位组成固定，不会因中途修改配置或重新照亮生成点而变种。
- `Enemy` 实例保存自身 `damage`、`attackRange`、速度和体型；守卫受击、建筑受击和渲染均使用实例属性。破坏者通过 `enemyRouteBlocker()` 优先选取路径上的 `defense` 类建筑。

## 人口生产与驻扎

- `nursery` 和 `training_ground` 通过 `recruits` 字段共享建筑内队列逻辑：完成后从建筑中心生成村民或守卫，并自动寻找对应类型可入住的建筑。
- 产房拥有两个临时生产岗位。玩家必须从真正空闲的村民中手动安排两人；招募队列只有在两人都抵达并处于 `WORKING` 时才累计进度，缺员时保留已有进度并暂停。每完成一名新村民，`releaseNurseryWorkers()` 会清空两个岗位并让原工人恢复 `IDLE`，后续队列需要重新安排人员。
- `G.maxPop` 与 `G.maxGuards` 分别限制村民和守卫。`availableRecruitSlots(role)` 会连同同类型的排队数量一起计算，避免多座产房或练兵房超额排队。
- `findNearestHome(pos, isGuard)` 按身份筛选床位：村民仅能入住大本营或住房，守卫仅能入住大本营或兵营。大本营基础提供 4 村民床位和 `MAX_GUARD_BASE`（默认 2）守卫床位。
- 左键点击或拖拽框选可见守卫；按 `E` 会对当前选中组统一切换自动/操控模式。操控模式会解除箭塔驻守，右键短按敌人下达攻击命令、右键短按地面下达按行进方向排列且共享主路径的编队移动命令，右键拖动仍用于移动镜头。
- 每次编队移动拥有独立的 `formationCommandId`。同一指令中仍在前往槽位的守卫，会忽略已经到位队友的局部避让和单位碰撞，避免近端槽位先站满后堵住远端槽位；不同指令、居民及仍在移动的队友继续正常碰撞，全部到位后恢复常规单位碰撞。
- 守卫攻击敌人时会建立仇恨目标。敌人会追击并攻击该守卫；距离超过 `ENEMY_GUARD_LEASH` 后清除仇恨，继续按原有逻辑推进和攻击建筑。
## 调试无限资源

- 点击“无限资源”后，大本营会获得所有资源各 `9999` 的真实仓储容量与库存，包括当前等级正常容量为 `0` 的资源。
- 无限资源开启期间，逻辑帧会持续补满大本营的实体库存；工程师取料、搬运和建造均读取这份真实库存，而不是只修改顶部资源统计。
- 仓储空余量仍按容量计算，不使用无限空余量，避免生产和搬运把库存写到容量上限之外。

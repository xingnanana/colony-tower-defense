# 技术实现

## 技术栈

纯原生 Web 技术，单文件架构，无框架依赖。

| 层级 | 技术 |
|------|------|
| 结构 | HTML5 |
| 样式 | CSS3（Flexbox、绝对定位） |
| 逻辑 | JavaScript ES6+（无 TypeScript、无 Babel） |
| 渲染 | Canvas 2D API |
| 存储 | 运行时内存（游戏存档未实现） |
| 文件 IO | Blob 下载导出 JSON + FileReader 导入 JSON |

## 架构模式

- **全局状态对象** `G`：所有运行时数据（建筑数组、居民数组、资源、相机等）
- **配置常量** `CFG`：世界尺寸、昼夜时长、速度数值等调参
- **面向对象**：`Building`、`Resident`、`Enemy`、`Particle` 四个核心类
- **游戏循环**：`requestAnimationFrame` 驱动的 `update(dt)` → `render()`
- **时间步长**：`dt` 基于 `performance.now()` 差值，上限 0.1s，乘以 `gameSpeed`
- **渲染分层**：世界空间（相机变换）→ 屏幕空间（HUD、时钟）

## 渲染系统

### 界面布局

- 页面使用三栏 CSS Grid：开发控制栏 168px、地图自适应、建筑目录 188px
- 建筑目录按设施、生产、仓储、防御分区，并使用 `bld-mark` 中文单字徽标；地板、住房不显示徽标
- 顶部资源栏使用 CSS `resource-mark` 颜色和几何形状区分资源类型
- 地图右下角工具条承载林务和施工班组操作，减少对地图主体的遮挡

### 相机变换
```
ctx.save()
ctx.translate(CANVAS_W/2, CANVAS_H/2)  // 原点移到画布中心
ctx.scale(zoom, zoom)                    // 缩放
ctx.translate(-cam.x, -cam.y)            // 平移到世界坐标
// ... 绘制世界内容 ...
ctx.restore()
// ... 绘制屏幕空间UI（时钟、夜间遮罩）...
```

### 世界坐标 ↔ 屏幕坐标
```
screenToWorld(sx, sy) → { x: (sx - W/2) / zoom + cam.x, y: (sy - H/2) / zoom + cam.y }
worldToScreen(wx, wy) → { x: (wx - cam.x) * zoom + W/2, y: (wy - cam.y) * zoom + H/2 }
```

### 建筑绘制
- 占地：虚线矩形（`setLineDash([3,6])`）
- 碰撞体：实心圆（`arc`），半径 = `min(w,h)/2 - 6`
- 蓝图：`globalAlpha = 0.5`
- 建造进度：自下而上填充矩形 + 百分比文字
- 建筑主体使用 Canvas 几何细节绘制，建筑代码仅作为低对比度识别辅助
- 大本营不绘制底部容量/住户信息条；所有建筑血条仅在受伤时绘制

### 居民绘制
- 碰撞体：半透明圆（半径 7px）
- 身体：椭圆（`scale(0.5, 1.0)` + `arc(0,0,10)`）→ 5×10px 半径 = 10×20px 直径
- 方向眼、携带物指示、饥饿/困倦图标
- 守卫使用盾牌轮廓，饥饿/困倦使用短条和警示标记，不使用 emoji

## 碰撞系统

`resolveCollisions(r, nx, ny)` 在每次移动前调用：

1. **建筑碰撞**：跳过道路、仓库/大本营（`isStorage`）、自己工作/居住建筑
   - 圆对圆：`dist < building.collisionRadius + RESIDENT_RADIUS` → 推开
2. **居民碰撞**：跳过 hidden 居民
   - 推开（50% 重叠量）+ 垂直错开（30% 重叠量）避免死锁
3. **世界边界**：`clamp(nx, RESIDENT_RADIUS, WORLD_W - RESIDENT_RADIUS)`

## 移动系统

- 当前采用目标点直线移动，通过 `resolveCollisions` 处理建筑和居民碰撞
- 地板格提供 1.5 倍移动速度
- A*/流场和道路系统当前均未启用
- 复杂建筑布局可能出现绕障失败，是后续需要单独设计的系统

## 昼夜系统

```
一天 = DAY_DURATION(162s) + TRANSITION(8s) + NIGHT_DURATION(78s) + TRANSITION(8s) = 256s
时钟时间 = (totalTime / 256 * 24 + 6) % 24  // 6:00 AM 起始
```

相位转换：`day → dusk → night → dawn → day`
过渡期间 `nightAlpha` 从 0 线性过渡到 0.55

## 资源系统

- 每个仓库的 `stored` 是库存事实来源，`G.resources` 是所有有效仓库的汇总缓存
- `depositToStorage` / `withdrawFromStorage` 统一更新具体库存和汇总缓存
- 专用仓库容量 50，大本营总容量 200，容量按仓库全部资源合计
- 工程师取料、居民吃饭、升级和炼铁都从实际仓库扣除
- `isStorage(type)` = `cat === 'storage' || type === 'town_hall'`
- `findNearestStorage(pos, resType, options)` 可按资源数量或剩余空间筛选

## 生产系统

- 生产进度在 `updateBuildings` 中每座建筑每帧结算一次
- `pendingOutput` 是生产建筑内部产物缓存，容量由 `CFG.PRODUCTION_BATCH_SIZE` 控制为 5；达到容量后由 `outputHauler` 搬运整批产物
- 批量搬运要求目标仓库一次性拥有完整批次空间；搬运中的居民不计入 `activeWorkersByBuilding`
- 护林员使用同样的 5 个批次规则连续砍树，树木不足时允许部分批次收尾
- 普通生产批次通过 `Resident.carryingFrom` 记录来源建筑；搬运途中居民被移除时，未送达部分退回建筑缓存。入库以实际存入量为准，剩余部分继续搬运。
- 黄昏不会打断携带物的搬运；黎明时仍携带物资的居民会恢复 `HAULING`，避免生产批次锁死。
- 效率由当前处于 `WORKING` 状态的在岗工人数决定，搬运中的工人不计入
- 农场连携在建筑布局改变时缓存，避免每帧重复遍历
- 采石场和铁矿通过 `findResourceNode` 校验附近资源点，并持续生产
- 炼铁机配方为 1 铁矿生产 1 铁锭

## 建筑交互点

- 生产工人前往生产建筑中心附近并通过建筑碰撞半径判定到岗。
- `Building.guardSpot()` 定义箭塔守卫的固定驻守位置；生产、仓储和施工目标使用建筑中心。以上目标点仍由现有连续移动和碰撞系统处理，不引入网格化行走。

## 性能考虑

- Canvas 渲染为主，HTML overlay 为辅
- 地板、网格、建筑、居民、敌人、资源和粒子按相机视口裁剪
- HUD 每 150ms 更新一次，避免每帧写 DOM
- 农场连携只在放置、移动、完成或拆除时重算
- 居民碰撞仍是全量遍历；人口达到数百后应改为空间哈希
- 粒子系统有生命周期自动清理

## 当前移动避障

- `flowMove()` 先检测当前位置到目标的直线是否穿过建筑碰撞范围。
- 直线受阻时，在阻挡建筑周围生成有限数量的连续坐标绕行点，选择当前可见且路径代价较低的一侧。
- 每个绕行点的出发段都会检查全部建筑、石矿和铁矿；到达该点后如仍有阻挡，会继续生成下一段局部绕行路线。
- 到达绕行点后恢复直达目标；目标改变或建筑失效时清除旧绕行点。
- 石矿和铁矿通过资源碰撞体阻挡移动，并参与绕行检测；树木和树苗不阻挡移动。
- `G.navigationRevision` 在建筑新增、移动、拆除或摧毁时递增；单位发现版本变化后立即重算路线。
- 单位每 `CFG.NAV_RECHECK_INTERVAL`（0.25 秒）验证当前两段路线，连续卡住 `CFG.NAV_STUCK_TIMEOUT`（0.5 秒）时强制重算。
- 居民之间仍使用 `resolveCollisions()` 做局部推开；当前方案不是网格寻路或完整 NavMesh，适合现有测试地图，复杂大地图仍需后续升级。

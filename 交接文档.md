# 村庄模拟塔防游戏 — 交接文档

## 项目概况

- **项目名**: 村庄模拟塔防游戏
- **入口**: `game.html` 只维护页面结构和脚本顺序；样式位于 `styles/game.css`，运行代码按领域拆分在 `js/`
- **技术栈**: 纯 HTML5 Canvas + Vanilla JS（无框架）
- **仓库**: `github.com/xingnanana/colony-tower-defense`
- **设计参考**: Kingdoms and Castles

### 人口系统
- **村民**: 产房招募（5食物/10秒），住房和大本营提供床位
- **守卫**: 练兵房招募（20食物/15秒），兵营和大本营提供守卫床位
- **初始**: `START_POP` 名村民，其中 `START_ENGINEERS` 名默认工程师；另有1名守卫
- 村民与守卫上限分离；大本营每级通过 `popBonus` / `guardBonus` 独立提供两类容量，住房和兵营分别追加村民与守卫床位。当前项目一级大本营配置为 3 村民上限、1 守卫上限

### 守卫系统
- HP=80，攻击=12/0.6s，搜索范围 450px
- 不吃饭不睡觉，白天隐藏睡觉，夜晚出来
- AI: 被分配箭塔的守卫前往操控；未分配守卫机动战斗 → 受伤回大本营回血
- 巡逻范围：所有建筑外接圆 + 100px
- 敌人攻击建筑时，巡逻守卫优先冲过去打

### 永久昼夜调试
- 调试面板保留“跳至白天/跳至夜晚”，并新增“永久白天/永久黑夜”。`day` 模式完整推进白天和定时用餐，日落时跳到次日早晨；`night` 模式完整推进夜晚，天亮时跳到下一夜并重新生成夜袭。两种模式都不会冻结生产、移动、动画或战斗
- 两个锁定按钮互斥且以高亮表示当前模式；再次点击当前按钮或使用普通昼夜跳转会解除锁定。重新开始和读取存档也会重置为正常昼夜循环
- 游戏内一天采用 12 小时时标。右上角钟面显示 `12、3、6、9`；“全局数值 → 时间与居民作息”以 `DAY_LENGTH` 配置一天总时长，并用 `0-11.5` 的双滑块配置 `NIGHT_START_HOUR` / `NIGHT_END_HOUR`。`DAY_DURATION` / `NIGHT_DURATION` 由 `syncDayNightDurations()` 派生；版本 1 的 24 小时配置在加载时自动除以 2，当前项目配置为夜晚 10:30 ~ 4:30、用餐 5:30 与 8:30。
- `tools/dev-server.js` 的 `/api/balance` 保存校验已同步到数值配置版本 2，同时保留版本 1 兼容。修改版本时必须同步客户端 `BALANCE_VERSION` 与服务端 `balanceVersion`，否则“保存到项目”会被拒绝。

### 地板系统（新增）
- 5×5格，1木/块，单击放置
- 独立 `G.floorMask` 网格（不在 buildings 中）
- 默认 1.5 倍移速加成，可在地板建筑配置中调整；地板没有实体等级，上限固定为 1
- 可放在任何位置（空地/建筑上/已有地板上）
- 底层渲染（地面层级）

### 当前界面与美术
- 游戏界面采用左侧开发控制栏（168px）、中央地图、右侧建筑目录（188px）的三栏布局
- 建筑目录按基础、生产、仓储、防御分区，生产/仓储/防御使用中文单字徽标；地板、住房和大本营不显示标识
- 顶部资源栏使用 CSS 颜色与形状标记资源、人口、守卫和昼夜
- Canvas 建筑使用几何结构绘制，资源点使用树木和矿石图形绘制
- 施工班组和林务操作集中在地图右下角，状态提示使用中文文字与颜色
- 游戏设置新增音效开关和 `0%` 至 `100%` 音量滑杆，默认启用且为 `55%`，修改后与快捷键设置一同自动保存在当前浏览器
- `js/audio.js` 使用 Web Audio 合成 UI、建造、生产、伐木、战斗、守卫指令和昼夜音效；世界声源按镜头距离衰减，高频同类事件有限频保护
- 伐木期间按树木摇晃节拍每 `0.55` 秒播放一次砍击声，砍倒后播放独立倒树声

### 生产与仓储
- 生产进度按建筑统一结算，不再由每名工人重复推进
- 效率按当前在岗工人计算，搬运中的工人不计入生产效率
- 生产建筑产物在建筑内部暂存，容量为 5；满批后由一名工人搬运，搬运者不计入在岗效率；批次可跨多个同类仓库拆分入库
- 护林员连续砍树并批量携带木材，附近没有下一棵树时允许携带不足 5 个收尾搬运
- 仓满不会阻塞搬运者或停止生产、伐木；无法入库的全部资源或部分余量会在搬运者当前位置形成同类地面物资，并释放原生产建筑的搬运锁
- 仓库恢复空位后，白天无携带物的非工程师村民会优先回收地面物资；空闲村民和暂时处于巡逻、返岗、生产状态的工人都可执行回收
- 普通生产批次记录来源建筑；搬运未完成时不会因仓库竞争、昼夜切换或居民移除而静默丢失
- 仓库容量真实生效；全局资源栏是各仓库库存汇总
- 居民吃饭、工程师取料、建筑消耗统一扣除具体仓库库存
- 采石场/铁矿必须建在对应矿点 3 格半径内，资源点可持续生产
- 炼铁机每生产 1 铁锭消耗 1 铁矿
- 只有树木允许居民直接砍伐；石矿和铁矿不能被手工采集，必须由对应矿场生产

### 护林员与农场连携
- 护林员小屋在范围空格内自动补种树苗，不需要工人在场；范围被占满时暂停，出现空缺后恢复补种
- 树苗基础成长 5 秒并带轻微随机波动；并行树苗数按满员砍伐速度推导，成熟后归属对应护林员小屋
- 护林员工人只负责砍伐成熟树与批量搬运木材
- 农场四面贴邻每面 +15% 产量，并显示连携线

### 砍伐系统
- 🪓砍伐 / ❌取消框选按钮（右下角 engineer bar 左侧）
- 框选标记树木 → 空闲村民自动砍；普通树和成熟果树共用 `TREE_CHOP_TIME` 与 `TREE_WOOD_YIELD`，默认 3 秒/棵、产出 5 木材
- 配置编辑器将“树木砍伐”与“果树种植与收获”分开；护林员建筑页不再重复配置砍伐时间，但“每级伐木效率”仍会加速所属工人
- 砍完自动搬运到仓库 → 找下一棵标记树
- 多人不重复砍同一棵树
- `marked` 标记控制，取消后村民立即放弃

### 地图自然资源
- `G.resourceNodes[]`：树默认 150 棵、石头默认 25 簇、铁矿默认 15 簇。
- 以大本营为圆心：树木半径 4 格外、石头 10 格外、铁矿 20 格外；初始可见范围至少有 5 棵树。
- 树簇为 1 至 10 棵、目标平均 4；石簇为 1 至 6 块、目标平均 1.5；铁矿通常单个但低概率形成 2 至 4 块小簇。全部参数位于“初始地图资源”配置分类，重开生效。
- 石矿和铁矿是移动障碍并参与绕行检测；树木和树苗不阻挡移动
- 资源在 `isCellBlocked` 中阻止建筑放置
- 树: 1×1，石/铁: 各占 1 格

### 寻路系统
- 直线可达时连续坐标直达；受阻时使用细格网 8 方向 A*、路径压缩、局部动态避让和停滞重算
- `resolveCollisions` 使用圆形建筑碰撞；农场与废墟不阻挡，施工目标不再因 `buildTarget` 身份被跳过
- 工程师通过 `buildingInteractionPoint()` 在蓝图、施工建筑或待修建筑的圆周外选择可达工作点；当前侧被相邻建筑封住时会改从其他方向接近
- 大本营及其他住所/岗位始终属于障碍；`home`/`workplace` 仅在单位起点仍处于其碰撞圈内时允许向外离开。例外是生产岗位工人在返岗、工作和送入原料时可进入自己的岗位建筑，离岗或其他单位再次经过仍须正常绕行
- 旧流场/Dijkstra 和道路优先方案已移除

### 迷雾系统
- 已实现仅“清晰/完全迷雾”的两态规则，不保留薄雾或上次探索状态。
- 大本营半径 8 格、灯半径 5 格；其他完成建筑仅点亮自身外接圆。居民半径 2 格、守卫半径 3 格，均为动态视野。
- `G.fogVisible` 每 0.2 秒重算，迷雾遮挡所有世界图层；迷雾中禁止建造、铺地和伐木标记。
- 敌人在不可见区域不渲染，也不会被箭塔选作攻击目标。后续关键地点探索标注尚未实现。
- 入夜时会按玩家当前视野的轮廓，预先选择距离清晰区至少 2 格的最近迷雾候选点。整夜队列不再扫描或换点，预定点被照亮时只延迟对应敌人。
- 夜袭总量会拆成多波，波内敌人在持续时间内随机错峰出现；后期增加波数，不显示“第 X 波来袭”提示。
- 箭塔与自动箭塔发射实体箭矢，伤害在箭矢命中时结算；箭矢或目标进入迷雾后不会继续造成伤害。
- 配置编辑器新增“敌人生成与波次”分类，数值会保存至 `data/balance.json`，并在下一次进入夜晚时使用。
- 配置编辑器的“建筑”分类已改为按建筑类型显示专属字段：灯编辑迷雾驱散范围，护林员/矿场编辑覆盖范围，生产建筑编辑工人与生产周期，仓库编辑容量，塔编辑战斗参数；共有的造价、耐久和建造时间按需显示，大本营因开局即存在而不显示建造时间。
- 建筑配置新增等级下拉与独立 `maxLevel`：可对每个等级写入 `levels[level]` 参数覆盖，未填写字段沿用既有默认升级曲线。大本营等级 1 额外配置迷雾范围、开局资源和五类资源的独立仓容；开局资源不再由全局配置分类管理。
- 原“全局数值”中的建筑成长参数已迁移：生产建筑配置批量、效率和暂存，护林员配置伐木与树苗参数，仓库配置扩容，箭塔/灯/城墙配置各自成长值。全局页不再保留这些建筑专属字段。

## 居民状态机

```
SLEEPING → (天亮) → IDLE/PATROL
IDLE/PATROL:
  - 工程师有蓝图 → GATHERING → BUILDING → CONSTRUCTING
  - 有工作 → GOING_TO_WORK → WORKING → HAULING → 循环
  - 非工程师有标记树 → GOING_TO_CHOP → CHOPPING(`TREE_CHOP_TIME`) → HAULING
  - 无 → 巡逻
GOING_TO_EAT → EATING（优先）
GOING_HOME → SLEEPING（优先）
```

## 守卫状态机

```
白天: GUARD_SLEEPING (hidden)
黄昏: GUARD_SLEEPING → GUARD_FIND_TOWER
  GUARD_FIND_TOWER:
    已分配箭塔 → GUARD_MANNING（塔启动攻击）
    未分配 → GUARD_FIGHTING（寻敌肉搏）
  GUARD_FIGHTING:
    HP ≤ 0 → GUARD_RETURNING → GUARD_HEALING(2h) → GUARD_FIGHTING
黎明: 任意状态 → GUARD_SLEEPING（回满血）
```

- 恢复餐上限由 `residentMealCost()` 计算为 `1 + missedMeals`，并通过 `withdrawFromAnyStorage()` 跨多个食物仓库扣除当前可支付数量。第一份食物支付当前餐，`completeResidentMeal(r, foodConsumed)` 再以 `foodConsumed - 1` 清除历史饥饿层数；库存不足允许部分恢复，库存为 0 才记录新的缺餐。

## 代码规范

- 文件: 使用无构建步骤的经典脚本分层；`config → world → navigation → tasks → entities → save → simulation/combat/residents/buildings/controls → developer UI → render/UI/input → main` 为固定加载顺序
- 风格: 紧凑无注释，函数内联
- 命名: `bl` (building), `r` (resident), `def` (BLD_DEFS entry)
- 建筑定义: `BLD_DEFS` 对象，key 为类型名
- 碰撞: `RESIDENT_RADIUS = 7`
- 建筑交互: 仓库取料/入库、回家、守卫回营和工程师施工均以圆形碰撞体外沿的可达交互点为移动目标，不得重新改为建筑中心目标；到达判定统一使用 `residentReachedBuilding()`，必须保留“进入外沿范围或抵达缓存交互点”的双重条件，以及圆周候选失败时向开放导航格吸附的保底逻辑。
- 工程师增员: `getAssignableResidents()` 同时返回无岗位漫游者和独立手工伐木者。伐木者使用 `pendingEngineer` 延迟转职，当前木材收尾后再设为 `isEngineer`；该字段会随游戏存档保存。
- 生产岗位碰撞: 仅本建筑岗位工人能进入机器碰撞体，返岗必须通过 `workplaceTravelPoint()` 先到可达外沿，再进入 `workplaceInteriorPoint()` 动态内部站位；不得让 A* 直接规划到被阻挡的内部格，也不得允许非本建筑人员穿透。
- 岗位候选: 工程师、产房及所有生产建筑统一使用 `getAssignableResidents()`。`residentHasFiniteIndependentTask()` 覆盖伐木、狩猎、果树种植、地面物资回收和独立搬运；这些居民分配后先收尾再上岗，禁止为产房重新建立只接受 `IDLE/PATROL` 的特殊候选池。
- 果树取料: `GOING_TO_PLANT_MATERIAL` 先由 `setBuildingInteractionTarget()` 写入外沿目标，移动必须使用 `r.targetX/r.targetY`，不得重新引用局部 `center`。
- 文档同步: 遵循 `文档维护规范.md`；代码、交接、README、术语、技术实现、TODO 和测试清单须在同一批改动中同步
- 界面图标: 游戏内使用 CSS/Canvas 几何徽标，emoji 仅允许出现在非界面说明文档中
- 大本营视觉: 不显示底部仓储容量条和住户条；建筑血条仅在当前 HP 低于最大 HP 时显示
- 仓库视觉: 所有专用仓库仅使用底部容量条显示库存与上限；任何建筑都不得重新显示 `BOX` 文字

## 运行方式

直接浏览器打开 `game.html` 仍可游玩；推荐开发时使用 3010 服务，以便加载并保存项目配置。

开发者调参时可在 VS Code 运行“Tasks: Run Task → 启动游戏开发服务器”（配置位于 `.vscode/tasks.json`），或直接执行 `node tools/dev-server.js`，访问 `http://127.0.0.1:3010/game.html`。配置编辑器可即时应用数值，并通过本地接口保存到 `data/balance.json`；刷新页面会自动加载保存值。直接打开 HTML 时保留游玩与临时编辑能力，但不能写项目文件。

`.vscode/launch.json` 的“调试村庄模拟（内嵌浏览器）”以该任务作为 `preLaunchTask`；在 VS Code 选择此调试配置后按 `F5`，会自动启动服务、打开 `editor-browser` 内嵌预览并启用 JavaScript 调试。

# 文档维护规范

本文件规定项目代码与文档的同步方式。任何功能、数值、交互或架构修改完成后，必须在同一批改动中更新对应文档。

## 权威来源

| 内容 | 权威来源 | 用途 |
|------|----------|------|
| 实际运行行为、数值、建筑字段 | `js/`、`data/balance.json` | 唯一代码事实来源；`game.html` 仅维护页面与加载顺序 |
| 当前已完成功能、已知问题、下一步 | `交接文档.md` | 开发交接基线 |
| 用户可见玩法与操作 | `README.md` | 玩家说明 |
| 术语、状态机、公式 | `术语规范.md` | 统一沟通口径 |
| 架构、数据流、性能策略 | `技术实现.md` | 实现说明 |
| 已确认设计演进 | `设计变化.md` | 决策历史 |
| 计划、优先级、完成状态 | `TODO.md` | 开发排期 |
| 回归检查步骤 | `测试清单.md` | 每次改动后的验证 |

当文档与代码冲突时，以 `js/` 运行逻辑和 `data/balance.json` 项目配置为准；修复文档不一致必须进入当前改动，不留到后续再处理。

## 修改同步矩阵

| 改动类型 | 必须同步 |
|----------|----------|
| 新增/删除建筑、资源、配方、数值 | README、交接文档、术语规范、TODO；重大设计同时更新设计变化 |
| 居民、守卫、敌人、生产或仓储规则 | README、交接文档、术语规范、技术实现、测试清单 |
| UI、操作、调试能力、美术反馈 | README、交接文档、测试清单 |
| 架构、渲染、寻路、性能策略 | 技术实现、交接文档、测试清单 |
| 修复 bug | 交接文档；若改变玩家可见行为，再同步 README 和测试清单 |
| 计划变更 | TODO、交接文档；若是设计方向调整，再同步设计变化 |

## 每次开发结束前

1. 对照本文件确认受影响文档已更新。
2. 更新 `TODO.md` 的完成状态与下一步优先级。
3. 在 `交接文档.md` 写清当前状态、限制和已知问题。
4. 按 `测试清单.md` 执行受影响系统的检查。
5. 运行语法检查，并记录无法完成的验证条件。

## 文档新增原则

只有当新内容需要长期独立维护、且无法清晰归入现有文档时才创建新文件。避免为同一份数值或规则维护多份表格；可配置的精确字段始终以 `js/config.js` 的 `CFG`、`BLD_DEFS` 与 `data/balance.json` 为准。
